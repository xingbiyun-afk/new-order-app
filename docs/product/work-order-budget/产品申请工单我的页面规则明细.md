# 产品申请工单我的页面规则明细（v0.2）

## 1. 文档说明

### 1.1 文档名称
产品申请工单我的页面规则明细

### 1.2 文档定位
本文档用于补充《产品申请工单和产品申请预算核销PRD》《产品申请工单字段清单》与现有 Vue 参考实现中，关于“我的”页的页面承接规则说明。

本文档重点用于明确：

- “我的”页在当前阶段的承接边界；
- 页面哪些部分沿用 App 现有工单通用架构；
- 当前需要重点细化的是哪些内容；
- 产品申请工单在“我的”页列表中的卡片展示口径；
- 搜索规则在产品申请工单场景下的补充说明；
- Mock 数据在“我的”页中的承接原则；
- 后续 003 变更清单与 AI 开发的参考依据。

### 1.3 当前阶段定位
当前阶段，“我的”页不重新设计页面整体骨架，而是在**复用 App 现有工单页结构**的前提下，细化产品申请工单在列表卡片内容、搜索口径与展示数据上的承接规则。

因此，本文档不是重新定义一个全新的“我的”页，而是用于明确：

- 哪些区域跟随 App 现有架构；
- 哪些内容需要针对产品申请工单单独明确；
- 哪些 mock 数据需要补充，以支撑页面验收与走查。

## 2. 页面整体承接边界

### 2.1 整体结构原则
“我的”页整体结构沿用 App 现有工单页面架构，不在当前阶段重新设计页面骨架。

当前默认沿用的区域包括：

1. 页面顶部标题区
2. 搜索输入区
3. 状态 Tab 区
4. 查询筛选区
5. 工单列表区
6. 底部 TabBar

### 2.2 本期不作为重点变更的区域
以下内容本期不作为产品申请工单规则细化的重点对象：

- 顶部标题区的布局与样式
- 搜索输入框的通用交互形态
- 查询筛选区的位置、结构与基础样式
- 底部 TabBar 的结构与交互
- 整体页面骨架与通用容器布局

上述内容统一遵循 App 现有工单页架构和通用交互规范，不在本文档中重新设计。

### 2.3 本期重点细化内容
本期“我的”页需重点明确的内容为：

1. 产品申请工单列表卡片展示哪些字段；
2. 卡片中哪些字段必须展示，哪些字段明确不展示；
3. Tab 页签与当前业务范围的关系；
4. 搜索规则在产品申请工单场景下如何承接；
5. 产品申请工单 Mock 数据在列表页中的展示原则；
6. 页面验收时需要具备的基础数据覆盖效果。

## 3. “我的”页范围说明

### 3.1 当前页签范围
当前“我的”页至少承接以下页签：

- 待办
- 已办
- 已发起

### 3.2 抄送我的
当前口径如下：

- `抄送我的` 本期**不纳入正式业务承接范围**
- 但在当前 Vue 参考实现中，**保留 Tab 视觉占位**
- 当前点击进入后，展示空态文案：
  - `暂无抄送工单`
- 本期不要求围绕“抄送我的”补完整业务规则闭环
- 后续是否正式纳入，另行通过变更清单明确

### 3.3 三个主要页签的承接定义
当前三大主页签的定义如下：

- `待办`：展示当前用户待处理的产品申请工单
- `已办`：展示当前用户已处理过的产品申请工单
- `已发起`：展示当前用户发起的产品申请工单

### 3.4 当前页面关注重点
当前“我的”页页面规则关注重点不在页签结构调整，而在：

- 不同页签下产品申请工单卡片如何展示；
- 当前参考实现是否有足够数量和分布合理的 mock 数据支撑走查；
- 页面是否能直观看到不同状态、不同字段内容和不同文本长度下的展示效果。

## 4. 列表卡片展示规则

### 4.1 卡片展示目标
“我的”页中的产品申请工单卡片，应以**移动端最小必要展示原则**承接，不追求一次性展示全部业务字段，而是以便于快速浏览、区分和进入详情页为目标。

### 4.2 卡片标题
卡片左上主标题固定展示为：

- `产品申请`

不展示：
- `产品申请工单`

### 4.3 卡片正式展示字段
当前产品申请工单列表卡片正式展示字段如下：

1. 工单类型：`产品申请`
2. 工单主状态
3. 申请人
4. 申请类型
5. 申请理由
6. 发起时间

### 4.4 卡片正文结构
卡片正文固定展示以下三行键值信息：

1. 申请人
2. 申请类型
3. 申请理由

不追加以下信息：
- 工单编号
- 预算号
- 订单归属专卖店
- 客户编号
- 客户名称
- 商品明细
- 金额合计
- 附件信息
- 审批备注

### 4.5 卡片状态展示
当前正式展示口径为：

- 处理中
- 已结束
- 已驳回

状态颜色语义固定如下：

- `处理中`：青绿色
- `已结束`：绿色
- `已驳回`：红色

### 4.6 申请理由展示规则
- 申请理由在卡片中按**单行**展示；
- 超出部分以省略号截断；
- 列表页不展开完整申请理由内容。

### 4.7 发起时间展示规则
- 列表卡片中展示字段为：发起时间
- 发起时间固定展示在卡片**右下角**
- 不与正文键值信息混排

发起时间格式为：

`月（00）-日（00） 时（00）-分（00）`

例如：
- `06-26 10:17`
- `06-25 14:30`

### 4.8 卡片明确不展示字段
当前列表卡片中明确不展示以下字段：

- 工单编号
- 预算号
- 订单归属专卖店
- 客户编号
- 客户名称
- 商品明细
- 金额合计
- 附件信息
- 审批备注

### 4.9 展示原因说明
当前卡片字段收敛原则如下：

- 工单编号不在列表卡片展示，避免列表信息过重；
- 预算号不在列表卡片展示，已按最新口径移除；
- 订单归属专卖店不在列表卡片展示，避免多分组结构下信息过长；
- 多专卖店申请分组、多商品明细等复杂信息，统一在详情页承接；
- 列表页重点用于快速浏览和进入详情，而非完整承载工单全部业务结构。

## 5. 搜索、筛选与跳转规则

### 5.1 搜索区整体原则
搜索区整体结构、位置、样式与基础交互方式，沿用 App 现有工单通用规则，本期不重新设计搜索区骨架。

### 5.2 搜索规则补充说明
在沿用现有工单搜索规则的基础上，针对产品申请工单，补充说明如下：

#### 5.2.1 发起人信息搜索
对产品申请工单而言，“发起人信息”即当前产品申请工单的发起人信息。

#### 5.2.2 专卖店 / 客户相关信息搜索
产品申请工单采用“工单头 + 多个专卖店申请分组”结构，一张工单中可能包含多个专卖店 / 客户信息。

当前搜索处理规则说明为：

- 与现有 App 工单搜索规则保持一致；
- 支持 **编号精确搜索**（数字）；
- 支持 **店名模糊搜索**（中文）；
- 支持 **客户名模糊搜索**（中文）。

补充说明：
- 若一张产品申请工单中包含多个专卖店申请分组，则只要命中该工单中任一分组对应的编号 / 店名 / 客户名，即应筛选出该工单。

#### 5.2.3 工单填报内容搜索
现有“工单填报内容”口径主要适用于客服、物流工单。  
对于产品申请工单，需补充承接为：

- 对`申请理由`进行模糊内容搜索。

即：
- 当搜索内容命中产品申请工单的申请理由时，应筛选出对应工单。

### 5.3 筛选区规则
筛选栏跟随 App 通用交互承接。

当前规则如下：

- 本文档不细化筛选项枚举；
- 本文档不细化排序实现细节；
- 仅保留结构说明；
- 具体筛选项与排序逻辑继续沿用现有工单通用实现。

### 5.4 卡片点击跳转规则
列表中的产品申请工单卡片，点击后统一进入详情页。

说明：
- 待办 / 已办 / 已发起 列表中的卡片点击规则一致；
- 列表页不单独设置“审批页”入口；
- 审批处理区由详情页按用户身份与工单状态条件展示承接。

## 6. 排序与列表承接边界

### 6.1 排序说明
排序逻辑沿用现有工单规则：

- 最新到达优先
- 最久等待优先

说明：
- 列表卡片显示的是发起时间；
- 但排序逻辑不要求严格按发起时间排序；
- 仍沿用现有工单“到达顺序 / 等待时长”逻辑承接。

### 6.2 列表承接方式
当前参考实现阶段，“我的”页列表采用滚动承接即可。

本期不在本文档中展开以下内容：

- 分页规则
- 加载更多规则
- 下拉刷新规则
- 数据分批加载规则

## 7. Mock 数据承接规则

### 7.1 当前 Mock 数据定位
“我的”页中的 mock 数据，当前主要用于：

1. 支撑页面结构验收；
2. 支撑卡片字段展示走查；
3. 支撑不同状态、不同文本长度、不同时间展示效果验证；
4. 为后续继续交给 AI 开发或调整样式提供可视化基础。

### 7.2 Mock 数据重点不是“有数据即可”
当前“我的”页的 mock 数据补充，应重点满足以下目标：

- 能看到足够数量的列表卡片；
- 能覆盖不同主状态；
- 能覆盖不同长度的申请理由；
- 能覆盖不同时间展示效果；
- 能支撑待办 / 已办 / 已发起的基本验收；
- `抄送我的` 作为视觉占位，当前以空态承接。

### 7.3 每个主要页签的建议数据量
当前主要页签默认建议展示：

- 待办：4 条
- 已办：4 条
- 已发起：4 条

说明：
- `抄送我的` 当前不要求补业务 mock 数据；
- `抄送我的` 当前以空态文案承接。

### 7.4 状态覆盖要求
“我的”页 mock 数据应覆盖以下三种主状态：

- 处理中
- 已结束
- 已驳回

并补充明确：

- `已办` 中需包含：
  - 已结束
  - 已驳回
- `已发起` 中允许同时出现：
  - 处理中
  - 已结束
  - 已驳回

### 7.5 文本长度覆盖要求
Mock 数据中的申请理由应至少覆盖：

- 短文本
- 中等长度文本
- 长文本（用于验证单行截断与省略号效果）

### 7.6 时间覆盖要求
Mock 数据中的发起时间应至少覆盖：

- 同一天不同时间
- 跨天时间展示

### 7.7 搜索覆盖说明
当前规则文档先明确搜索处理口径，不强制要求本轮 mock 数据必须完整覆盖所有搜索组合场景。  
后续是否针对编号精确搜索、店名模糊搜索、客户名模糊搜索进一步补足专项 mock 数据，可结合后续开发与验收情况再细化。

## 8. 空态规则

### 8.1 抄送我的
当前空态文案固定为：

- `暂无抄送工单`

### 8.2 其他页签通用空态
当待办、已办、已发起任一页签无数据时，当前参考实现阶段可采用简单文字空态：

- `暂无工单`

说明：
- 当前阶段不要求设计复杂空态插画或重型占位组件；
- 以简单文字承接即可。

“我的”页在当前阶段的验收重点包括：

1. 页面整体结构是否与 App 现有工单架构保持一致；

3. 产品申请工单卡片字段是否与现行规则一致；
4. 卡片标题是否固定为“产品申请”；
5. 卡片正文是否固定为三行；
6. 申请理由是否按单行 + 省略号展示；
7. 发起时间是否位于右下角；

9. 是否已去掉不应出现在卡片中的字段（如预算号）；
10. 搜索规则在产品申请工单场景下是否已补充说明；

12. “抄送我的”是否仅作为视觉占位并正确展示空态。

   - 申请人

7. 卡片正式展示字段为：
   - 工单类型
   - 工单主状态
   - 申请人
   - 申请类型
   - 申请理由
   - 发起时间
8. 卡片明确不展示：
   - 工单编号
   - 预算号

   - `暂无抄送工单`
10. 搜索规则整体沿用现有工单通用规则，并补充产品申请工单的发起人、专卖店 / 客户信息、申请理由搜索说明；
