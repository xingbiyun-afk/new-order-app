# 产品申请工单审批承接与异常处理说明（v0.1）

## 1. 文档说明

### 1.1 文档名称
产品申请工单审批承接与异常处理说明

### 1.2 文档定位
本文档用于说明 **App 端产品申请工单** 对以下能力的承接关系：

- 产品申请审批流程管理
- 审批异常监控
- 审批超时自动驳回
- 组织变更场景在途工单处理
- 详情页状态 / 驳回原因 / 审批流展示承接

本文档是 **App 仓的承接型规则文档**，不是后台 Web 仓的正式页面主规则文档。

### 1.3 适用范围
本文档适用于：

- `vue-reference/` 中产品申请工单详情页审批相关展示规则
- 产品申请工单状态承接规则中与审批异常相关的边界
- AI 执行详情页改造、mock 修正时的审批异常边界规则依据

### 1.4 与其他文档的关系
本文档基于以下文档和变更结论整理：

- 《产品申请工单流程与状态规则》§10-§13
- 《产品申请工单详情页面规则明细》§5.4、§6.2.3-§6.2.4、§6.9.3
- `CR-20260709-003 审批超时自动驳回、组织变更场景在途工单驳回及详情承接规则补增`

### 1.5 跨仓归属说明
| 能力 | 正式归属仓库 | 当前 App 仓角色 |
|------|-------------|----------------|
| 产品申请审批流程管理 | `new-order-system`（后台 Web 仓） | 承接侧：仅展示审批流结果 |
| 审批人员异常监控 | `new-order-system`（后台 Web 仓） | 承接侧：仅消费最终审批结果 |
| 审批超时自动驳回 | `new-order-system`（后台 Web 仓） | 承接侧：仅展示系统驳回结果 |
| 组织变更场景在途工单处理 | `new-order-system`（后台 Web 仓） | 承接侧：仅展示最终驳回结果 |

> **核心原则**：本仓只沉淀与产品申请工单 App 展示和状态承接直接相关的最小必要规则。若后台仓后续形成正式审批流程管理 / 异常监控文档，以后台仓正式文档为准，本仓文档负责 App 侧承接说明，不重复维护后台页面完整规则。

## 2. 正式归属与承接边界

### 2.1 后台仓正式归属
- 后台 Web 正式能力归属：`new-order-system`
- 产品申请审批流程管理的配置页面在后台仓
- 审批人员异常监控的监控页面在后台仓

### 2.2 App 仓不承接的内容
1. 后台审批流程管理配置页面规则
2. 后台审批异常监控页面规则
3. 审批节点配置细节
4. 监控提示 UI 设计
5. 定时任务 / 调度实现

### 2.3 App 仓仅承接的内容
1. **工单状态承接**：处理中 / 已驳回 / 已结束
2. **详情页展示承接**：驳回信息区、审批流区、驳回原因分类
3. **驳回来源分类**：4 类固定分类的展示口径
4. **监控 / 通知 / 系统处理之间的边界**：明确哪些是监控提示、哪些是系统结果

### 2.4 不在本仓重复定义的内容
- 后台审批流程管理的完整产品设计
- 后台审批异常监控的完整产品设计
- 接口协议定义
- 系统任务 / 调度设计

## 3. 审批流程管理承接口径

### 3.1 基本规则
发起人与审批人统一来源于**产品申请审批流程管理**。

### 3.2 对工单的影响
1. 工单提交后按审批流程管理配置进入对应审批节点
2. 详情页审批流展示的数据来源于审批流程管理结果
3. 当前审批进度区展示的"当前待处理节点 / 当前待处理人"来源于审批流程管理

### 3.3 App 仓承接边界
- 本仓只承接"对工单展示的影响"
- 不展开后台配置页细节
- 审批流节点列表 `approvalNodes` 的数据结构与审批流程管理的节点配置对应，但本仓只定义展示口径，不定义配置口径

## 4. 审批超时自动驳回规则

### 4.1 适用范围
产品申请工单审批中的**任一审批环节**。

### 4.2 超时阈值
**14 天**。

### 4.3 起算点
自**进入当前待审批节点时刻**起算。

### 4.4 触发条件
连续超过 **14 天**无审批操作（同意 / 驳回 / 其他有效审批处理）。

> 若节点期间已发生审批动作，则重新起算。

### 4.5 处理结果
1. 系统自动驳回后，工单主状态承接为：**已驳回**
2. 自动驳回属于**系统处理结果**，不是人工审批意见
3. 详情页驳回原因中需明确体现"审批超时自动驳回"语义

### 4.6 人工驳回 vs 系统自动驳回区分
| 维度 | 人工驳回 | 系统自动驳回 |
|------|----------|-------------|
| 执行者 | 审批人 | 系统 |
| 触发原因 | 业务判断 / 财务通知后执行 | 超时无操作 |
| 详情页展示 | 审批人姓名 + 驳回意见 | "系统超时自动驳回" + 超时说明 |
| 审批流展示 | 展示实际审批节点 + 审批人 | 展示"系统超时自动驳回" |

## 5. 组织变更场景在途工单处理规则

### 5.1 适用场景
工单已提交且处于审批中，相关组织信息发生变更。

### 5.2 处理流程
1. 对于变更前属于该组织范围的在途工单，由**财务通知相关环节审批人驳回**
2. 财务通知本身**不改变工单状态**
3. 通知后审批人执行驳回 → 按**人工驳回**落定
4. 通知后 14 天仍无人处理 → 按**系统自动驳回**兜底

### 5.3 约束
1. 组织变更场景下**不新增新的主状态**，最终仍统一归入：**已驳回**
2. 页面需体现"组织变更"是驳回背景原因之一
3. 不得把"财务已通知"或"已通知财务"展示成审批结果

## 6. 审批异常监控与系统处理边界

### 6.1 职责划分
| 职责方 | 负责内容 | 不负责内容 |
|--------|----------|------------|
| 审批异常监控 | 超时识别、监控提示、预警生成 | **不直接改变工单结果** |
| 工单系统 | 状态变更、详情页展示、驳回信息记录 | 不干预监控逻辑 |

### 6.2 核心原则
**监控不直接改变工单结果，只作为识别与推动依据。**

### 6.3 数据流
1. 监控层识别超时 → 生成提示/预警 → 推送给相关人员
2. 系统层判断超时 → 执行自动驳回 → 改变工单状态
3. 两者独立运作，监控层不干预系统层的执行逻辑

### 6.4 App 详情页承接
- App 详情页展示**最终审批结果**，不展示监控过程页面细节
- 驳回信息区展示驳回原因（含超时自动驳回）
- 审批流区展示节点结果（含系统驳回标识）
- 不展示"监控预警中"等中间状态

---

## 7. 详情页承接口径

### 7.1 主状态
主状态仍为以下 3 类，不新增：
- **处理中**
- **已驳回**
- **已结束**

### 7.2 驳回信息区承接
- 驳回原因按 §8 四类分类口径展示
- 系统超时自动驳回需明确标识为系统处理结果
- 组织变更场景需体现"组织变更"背景信息

### 7.3 审批流区承接
- 人工驳回：展示实际审批节点名称 + 审批人姓名
- 系统超时自动驳回：展示"系统超时自动驳回" + "系统"作为执行者
- 不得把系统超时自动驳回展示为某个审批人的审批意见

### 7.4 约束
- 不改动订单结果区主结构
- 不新增复杂状态树
- 驳回原因分类与订单结果说明是独立的两套口径

## 8. 驳回原因来源分类总表

详情页"已驳回"状态下，驳回原因按以下 **4 类分类**承接：

| 分类 | 触发场景 | 详情页展示口径 | 执行者 |
|------|----------|---------------|--------|
| **普通审批驳回** | 审批人基于业务判断主动驳回 | 驳回人 + 驳回意见 | 审批人 |
| **审批超时自动驳回** | 审批节点超过 14 天无操作 | "审批超时自动驳回：当前审批节点超过 14 天无审批操作" | 系统 |
| **组织变更后人工驳回** | 组织变更后，财务通知审批人执行驳回 | "因组织变更，按财务通知驳回" + 驳回人 + 驳回意见 | 审批人 |
| **组织变更后超时自动驳回** | 组织变更后仍无人处理，超 14 天 | "审批超时自动驳回：组织变更后超过 14 天无审批操作" | 系统 |

> 说明：
> - 上述 4 类属于**驳回原因 / 来源分类**
> - 不等同于新增主状态（主状态仍为"已驳回"）
> - 不等同于新增预算状态
> - 不等同于新增订单结果分类
