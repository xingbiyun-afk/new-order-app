## 2. 页面结构设计原则

### 2.1 复用现有工单交互
产品申请工单应尽量复用现有工单移动端交互模式，包括：

- 行式表单布局；
- 点击字段进入搜索/选择页；
- 搜索页输入关键字后展示结果列表；
- 选中结果后返回表单页回填；
- 我的列表页卡片式承载；
- 详情页分区块展示；
- 审批区与详情页共页承接。

### 2.2 优先保证结构闭环
当前阶段优先实现：
- 可发起
- 可查看列表
- 可查看详情
- 可审批
- 可查看结果

不要求当前阶段把所有异常场景都做成复杂交互。

### 2.3 移动端节省空间原则
考虑到发起页字段较多、产品明细为多条，且同一工单可包含多个产品申请订单明细，移动端页面需遵循以下原则：

- 能自动带出的字段不做可编辑输入；
- 折扣不单独占字段展示，仅保留“是否打折产品”；
- 产品明细与订单归属专卖店按分组组织；
- 同类信息尽量通过区块和分组收拢；
- 多条关联订单采用简化卡片式承载。

## 3. 核心结构模型

## 3.1 工单结构模型
产品申请工单不是“工单头 + 一组产品 + 一个专卖店”的结构，而是：

- 工单头信息
- 多个产品申请订单明细

结构示意：

```text
产品申请工单
  ├─ 预算号
  ├─ 申请人
  ├─ 申请类型
  ├─ 申请理由
  ├─ 附件
  └─ 产品申请订单明细列表
       ├─ 分组1：A专卖店
       │    ├─ SKU1
       │    ├─ SKU2
       │    └─ SKU3
       ├─ 分组2：B专卖店
       │    ├─ SKU4
       │    └─ SKU5
       └─ 分组3：C专卖店
            └─ SKU6
```

## 3.2 产品申请订单明细定义
产品申请订单明细定义如下：

1. 每个分组对应一个订单归属专卖店；
2. 每个分组下可维护多条产品 SKU 明细；
3. 一张产品申请工单可包含多个产品申请订单明细；
4. 不同专卖店的产品明细不能混在同一个分组中；
5. 功能订单和后续正式订单结果需能对应到具体产品申请订单明细。

## 5. 发起页结构草图说明

## 5.1 发起页整体结构
发起页建议采用以下区块顺序：

1. 页面标题区
2. 预算信息区
3. 申请基础信息区
4. 产品申请订单明细区
5. 附件上传区
6. 提交操作区

---

## 5.2 发起页区块结构

### 5.2.1 页面标题区
建议内容：
- 页面标题：产品申请工单
- 返回按钮
- 如现有工单页已有统一头部，则沿用现有样式

---

### 5.2.2 预算信息区
建议字段顺序：

1. 预算号
2. 预算异常提示（如有）

结构示意：

```text
[预算信息]
预算号                > 选择预算
异常提示（仅 isAbnormal=true 时展示）
```

说明：
- 点击预算号进入预算选择页；
- 预算选中后，联动带出申请类型、申请理由；
- 若预算异常但可用，仅在该区显示提示，不在详情页重复展示。

---

### 5.2.3 申请基础信息区（CR-20260701-001 两态优化）
建议字段顺序：

1. 申请人（始终展示）

**预算未选择前**：
- 区域压缩，仅展示申请人信息；
- 区域右上角提示"选择预算后展开"。

**预算选择后**：
- 展开展示完整的：
  1. 申请人
  2. 申请类型
  3. 申请理由

结构示意：

```text
(预算未选)
[申请信息            选择预算后展开]
申请人                张三

(预算已选)
[申请信息]
申请人                张三
申请类型              新品申请
申请理由              季度产品推广...
```

说明：
- 申请人自动带出，只读；
- 申请类型、申请理由由预算号带出，只读；
- 申请理由为多行展示；
- 预算未选时压缩视觉占用，提升首屏信息密度。

---

### 5.2.4 产品申请订单明细区
这是发起页最核心区块。  
页面不再采用“产品明细区 + 单独订单归属专卖店区”的结构，而改为：

- 产品申请订单明细列表
- 每个分组内包含：
  - 订单归属专卖店
  - 产品明细列表
  - 分组内金额小计（可选）
- 页面底部统一金额合计
- 支持新增分组
- 支持删除分组

## 5.4 分组区交互说明

### 5.4.1 新增分组
用户点击“添加专卖店分组”后，新增一个空分组。

空分组至少包含：
- 订单归属专卖店选择入口
- 空产品明细区
- 添加产品按钮

### 5.4.2 分组内产品维护
每个专卖店分组下：
- 支持新增多条产品
- 支持删除某条产品
- 每条产品独立计算金额
- 分组内产品只能归属当前分组所选专卖店

### 5.4.3 删除分组
支持删除整个产品申请订单明细。  
删除后，该分组下所有产品明细一并删除。

### 5.4.4 金额计算
- 每条产品计算金额（净额）
- 页面可选展示分组内金额小计
- 页面底部必须展示整张工单金额合计

## 5.5 分组内产品交互说明

### 5.5.1 单条产品交互流程
```text
在某个专卖店分组内
  ↓
点击选择产品编号
  ↓
带出产品名称 / JDE价格 / 是否打折产品 / 可申请数量
  ↓
输入数量
  ↓
自动计算金额（净额）
```

### 5.5.2 展示压缩建议
为了节省移动端空间：
- 不展示独立折扣字段；
- 仅展示“是否打折产品：是 / 否”；
- JDE价格、可申请数量、金额（净额）采用紧凑行式展示；
- 产品过多时允许后续增加折叠能力，但当前不是 P0 必做项。

### 5.5.3 数量控制说明
- 数量输入需受可申请数量前置控制；
- 提交时仍以后端库存校验结果为准；
- 同一 SKU 的总申请数量校验，需以实际业务校验口径和后端结果为准；
- 当前页面阶段重点完成基础前置控制和结果提示承接。

## 5.6 批量导入弹窗结构（CR-20260701-002）

### 5.6.1 入口位置
在发起页"产品申请订单明细区"的整单级总操作区域，与 `添加产品申请订单` 按钮并列展示 `批量导入产品` 按钮。

### 5.6.2 弹窗结构示意

```text
┌ 批量导入产品 ────────────────────────────┐
│                                           │
│  ┌─ 字段顺序说明（醒目区块）─────────┐    │
│  │ 字段顺序：产品编号，数量，专卖店编号  │  │
│  │ 支持逗号、空格、Tab 分隔，每行一条    │  │
│  └────────────────────────────────────┘    │
│                                           │
│  ┌─ 数据输入区 ─────────────────────┐    │
│  │ SKU001,10,ST0001                  │    │
│  │ SKU002,3,ST0001                   │    │
│  │ SKU003,8,ST0002                   │    │
│  │                                   │    │
│  └───────────────────────────────────┘    │
│                                           │
│  ┌─ 文件导入区 ─────────────────────┐    │
│  [📁 选择文件  ]    [📥 下载模板]     │    │
│  已选择：导入模板.csv                    │  │
│  └───────────────────────────────────┘    │
│                                           │
│  ┌─ 校验结果区（有错误时展示）──────┐    │
│  │ ⚠️ 第 3 行：产品编号不在核销范围    │  │
│  │ ⚠️ 第 5 行：专卖店编号不存在      │  │
│  │ ⚡ 第 7 行：数量超出可申请数量      │  │
│  └───────────────────────────────────┘    │
│                                           │
│  [取消]              [识别检查] [确认导入] │
└───────────────────────────────────────────┘
```

### 5.6.3 弹窗交互流程

```text
用户点击"批量导入产品"
  ↓
若页面已有数据 → 弹出覆盖确认
  ├─ 取消 → 返回发起页
  └─ 确认 → 清空 → 打开导入弹窗
  ↓
用户输入数据（粘贴/文件）
  ↓
点击"识别检查"
  ↓
系统解析并校验
  ├─ 存在硬拦截错误 → 展示全量错误 → 禁止导入
  ├─ 仅存在超量警告 → 展示警告 + 允许导入
  └─ 无错误 → 展示成功提示 + 允许导入
  ↓
点击"确认导入"
  ↓
按专卖店编号拆分生成多个产品申请订单明细
  ↓
关闭弹窗，发起页展示导入结果
```

### 5.6.4 覆盖导入弹窗

```text
┌ 提示 ─────────────────────────────────┐
│                                       │
│  批量导入后，将覆盖当前已填写的       │
│  产品申请订单明细和产品信息，         │
│  是否继续？                           │
│                                       │
│  [取消]                    [确认]     │
└───────────────────────────────────────┘
```

### 5.6.5 结构要点
1. 字段顺序说明区使用醒目样式（浅橙背景），帮助用户快速确认格式；
2. 数据输入区使用等宽字体，支持多行输入；
3. 文件导入区支持选择 Excel/CSV 文件，选择后自动解析填入输入区（用户仍可手工编辑）；
4. 模板下载提供 CSV 格式模板（UTF-8 BOM 编码）；
5. 校验结果区分硬拦截（⚠️）和软警告（⚡）；
6. 全有或全无策略：存在硬拦截错误时，不落页面，直接展示全量错误清单；
7. 确认导入后按专卖店编号自动拆分生成多个产品申请订单明细。

## 6. 预算选择页结构草图说明（CR-20260702-001 全面更新）

> **页面级详细规则**：本章节描述预算选择页面的结构布局与交互草图。详细的字段展示规则、标签计算规则、可选性判定规则、排序规则、冻结期与原单重提特例规则，以 `docs/product/work-order-budget/产品申请工单选择预算页面规则明细.md` 为准。

## 6.1 页面定位
预算选择页用于从当前用户可选预算中选择一个预算号，并返回发起页。
驳回后重提场景不进入此页面，预算号由系统锁定直接回填发起页。

## 6.2 页面结构

1. 页面标题区（含返回按钮）
2. 搜索栏（仅支持预算号模糊搜索）
3. 预算卡片列表区
4. 底部确认按钮区（选中预算后展示）

## 6.3 搜索栏结构

```text
[选择预算]                    [< 返回]

┌─────────────────────────────────────┐
│ 🔍 搜索预算号                        │
└─────────────────────────────────────┘
```

说明：
- 仅支持按预算号模糊搜索；
- 清空搜索词后展示全部可选预算；
- 搜索无结果时展示"搜索无结果"空态（区别于"当前无可申请预算"空态）。

## 9.4 不同状态下的结构重点（CR-20260706-001 新增）

详情页在三种状态下的阅读重点：

- **处理中**：状态、当前审批进度、发起背景、预算与申请信息、产品申请订单明细、附件、审批流、订单结果
- **已驳回**：驳回原因、驳回时间、驳回人、备注、是否允许重新发起 / 如何重新发起优先；其后再看发起背景、预算与申请信息、产品申请订单明细、附件、审批流
- **已结束**：最终状态、订单结果、失败原因优先，其后再看基础与过程信息

## 9.5 字段分区与层级（CR-20260706-001 新增）

详情页字段分区：

- **基础信息区**：工单号、当前状态、申请人、申请组织、创建时间
- **发起信息区**：申请类型、申请理由
- **预算与申请信息区**：预算号、预算归属组织、预算使用范围、预算总额（净额）、本次申请额度（净额）

层级要求：

- "本次申请额度（净额）"为主信息
- "预算总额（净额）"为辅助信息
- 不重复展示同一字段
- 不恢复"工单类型"等冗余字段

## 9.6 产品申请订单明细区结构（CR-20260706-001 新增）

详情页产品申请订单明细区按"汇总区 → 明细卡片头 → 展开区"三层结构组织：

- **汇总区**至少承接：

  - 涉及专卖店数量
  - 产品数量（可细化为款 / 件）
- **卡片头**至少承接：
  - 明细编号
  - 订单归属专卖店

- **展开区**承接：
  - 产品编号 / 产品名称

## 11. 订单结果区结构说明（CR-20260706-002 更新）

## 11.1 展示原则（CR-20260706-002）
- 正式订单结果为主信息；
- 预占库存订单（功能订单）属于过程信息，只展示订单编号，不展开复杂状态流转；
- 失败原因需贴近对应正式订单结果展示；
- 按产品申请订单明细承接结果，不把过程性订单信息和最终业务结果混在同一层级。

## 11.2 结果主次关系（CR-20260706-002）
1. **正式订单结果**（主信息）：订单类型、订单编号、订单状态
2. **失败原因**（贴近主信息）：多条失败原因通过数组去重后，以多个提示条展示，紧贴对应结果卡片
3. **预占库存订单编号**（辅助信息）：只展示编号，字号降级（12px、#999），不作为独立结果区块放大
4. **归并说明**（如适用）：当多笔明细归并到同一正式订单时，在结果卡片内说明归并关系

## 11.3 建议结构
每个产品申请订单明细结果卡片至少承接：

- 明细编号
- 订单归属专卖店（客户编号 + 客户名称）
- 预占库存订单编号列表（functionOrderNos，只展示编号，辅助字段）
- 正式订单结果卡片：
  - 订单类型（产品申请表订单 / 内部申请表订单）
  - 订单编号
  - 订单状态
  - 失败原因（如失败，贴近展示）
  - 归并说明（如多明细归并）

## 11.4 预占库存订单展示边界（CR-20260706-002）
1. 工单发起后生成预占库存订单，审批通过后或驳回后均取消，最终结果恒为"取消"；
2. 详情页中预占库存订单**只展示编号**，不展示状态、取消原因、详情跳转等复杂信息；
3. 一般按每一笔产品申请订单明细对应一个预占库存订单编号承接；
4. 展示层级明显低于正式订单结果；
5. 文案使用"预占订单"而非"功能订单"，避免误导为最终业务结果。

## 11.5 失败原因展示规则（CR-20260706-002）
1. 失败原因通过 `failReasons` 数组承接，展示前去重；
2. 每条失败原因紧贴对应正式订单结果卡片展示（内联提示条样式）；
3. 不允许把失败原因统一堆在区块底部；
4. 部分成功、部分失败场景下，成功结果与失败结果需能快速区分；
5. 失败原因文案尽量业务化，不只给技术判断结果。

## 11.6 归并展示规则（CR-20260706-002）
1. 同店编号的多笔产品申请订单明细，审批通过后归并到同一正式订单；
2. 归并结果需在对应结果卡片中说明："由明细 ①/② 归并生成"；
3. 归并说明不可喧宾夺主，但必须足够清楚；
4. 不支持只展示一个订单编号而不说明对应哪些明细。

## 14. 可直接交给 AI 的结构实现重点

当前页面结构说明已经足以支撑 AI 开始 Vue 参考实现，重点要求如下：

1. 复用现有工单移动端交互风格；
2. 发起页按“预算 → 申请信息 → 产品申请订单明细 → 附件 → 提交”顺序组织；
3. 产品申请订单明细为发起页核心结构；
4. 每个分组下包含订单归属专卖店和多条产品明细；
5. 产品明细采用分组内卡片/行式结构，不做复杂表格；
6. 折扣不单独展示，只展示是否打折产品；
7. 订单归属专卖店采用现有搜索选择页模式；
8. 列表卡片字段保持最小集；
9. 详情页与审批页共页承接；
10. 功能订单与关联订单结果按产品申请订单明细展示；
11. 审批操作区底部固定更优先；
12. 发起页提供整单级批量导入能力，支持粘贴识别和文件导入，按专卖店自动拆分；
13. 当前以结构、交互、样式参考实现为主，不要求一次完成完整生产联调。
