## 2. 页面适用对象与状态范围

### 2.1 页面适用对象
本页面适用于“产品申请工单”详情查看场景。

### 2.2 页面主状态范围
当前详情页正式承接以下 3 类主状态：

1. **处理中**
2. **已驳回**
3. **已结束**

### 2.3 状态表达约束
1. 页面层不定义正式状态枚举；
2. 状态展示统一通过展示字段承接，如：
   - `displayStatus`
   - `approvalStatusText`
   - `flowStatusText`
3. 页面规则以状态语义为准，不以某个临时字段命名硬绑定。

---

## 3. 页面整体结构规则

### 3.1 固定主结构顺序
详情页整体结构按以下顺序承接：

1. 页面标题区
2. 驳回信息区（仅已驳回时）
3. 当前审批进度区（仅处理中且存在当前节点时）
4. 基础信息区
5. 发起信息区
6. 预算与申请信息区
7. 产品申请订单明细区
8. 附件区
9. 审批流区
10. 订单结果区
11. 操作区

### 3.2 区块展示原则
1. 区块之间必须有明确标题或分隔；
2. 同层级区块标题样式应统一；
3. 页面首屏不应堆叠过多次级字段；
4. 若某区块当前状态下无有效信息，可不展示或弱化展示；
5. 不允许为了“结构完整”强行展示空壳区块。

### 3.3 页面阅读路径原则
详情页阅读路径需稳定为：

- 先看当前状态和结论；
- 再看发起背景与预算关系；
- 再看产品申请订单明细；
- 再看审批过程与结果；
- 最后看附件与辅助操作。

---

## 4. 状态承接规则

> 本章用于替代“顶部状态区必须始终存在”的旧写法。  
> 当前详情页状态承接应以**页面内容区统一表达**为准，不建议继续保留“仅部分状态在导航栏右上角显示状态字”的不一致方案。

### 4.1 当前统一规则
1. 当前阶段**不强制保留导航栏右上角独立状态字**；
2. 详情页主状态统一由页面内容区承接；
3. 基础信息区保留“当前状态”字段；
4. 不同状态再通过不同的重点区块强化表达：
   - 处理中：当前审批进度区
   - 已驳回：驳回信息区
   - 已结束：订单结果区

### 4.2 统一性要求
1. 不允许出现“处理中无右上角状态字，而已驳回 / 已结束有右上角状态字”的不一致承接；
2. 如后续要恢复顶部状态标签，必须三种状态统一设计后再纳入，不允许只给部分状态补标签；
3. 当前阶段优先采用“取消右上角独立状态字”的方案，避免和基础信息区“当前状态”重复。

### 4.3 当前建议方案
当前详情页推荐按以下口径收口：

- 导航栏仅展示页面标题，不单独展示状态字；
- 基础信息区中保留“当前状态”字段；
- 通过状态对应的重点区块承接主状态感知：
  - 处理中 → 当前审批进度区
  - 已驳回 → 驳回信息区
  - 已结束 → 订单结果区前置强化

## 5. 各状态下的页面重点规则

### 5.1 处理中

#### 5.1.1 页面重点
处理中场景下，页面重点应为：

1. 当前状态
2. 当前审批进度
3. 当前审批节点 / 审批流
4. 发起背景与预算申请信息
5. 产品申请订单明细
6. 附件
7. 订单结果区（如无正式结果，应不展示或仅占位弱化）

#### 5.1.2 核心规则
1. 处理中的页面核心问题是“当前到哪一步了”；
2. 当前审批进度区应前置承接；
3. 审批流区在处理中场景下权重高于已结束场景；
4. **处理中场景不得展示正式订单结果卡片**；
5. 处理中场景如需展示预占库存订单，仅允许在审批流发起节点承接，不在其他区块重复展示。

---

### 5.2 已驳回

#### 5.2.1 页面重点
已驳回场景下，页面重点应为：

1. 驳回信息
2. 是否允许重新发起 / 如何重新发起
3. 发起背景与预算申请信息
4. 产品申请订单明细
5. 附件
6. 审批流（突出驳回节点）
7. 历史过程信息（如有）

#### 5.2.2 核心规则
1. 驳回信息必须在页面中明显前置；
2. 用户进入页面后无需大幅滚动即可理解：
   - 为什么被驳回；
   - 谁驳回；
   - 什么时候驳回；
   - 下一步怎么办；
3. 审批流区需能识别驳回节点，但不能抢过驳回信息区；
4. 重新发起入口逻辑必须与创建页承接口径一致；
5. 不得引入“预算锁定 / 锁定状态 / 预算被锁定”等错误概念。

---

### 5.3 已结束

#### 5.3.1 页面重点
已结束场景下，页面重点应为：

1. 最终状态
2. 正式订单结果
3. 失败原因
4. 归并结果说明
5. 审批流
6. 附件
7. 预占订单编号（辅助信息）

#### 5.3.2 核心规则
1. 已结束场景首屏应优先承接“最终结果”；
2. 订单结果区层级应高于审批流区；
3. 如存在部分成功 / 部分失败，必须清晰区分；
4. 如存在重试历史，可在结果区中承接；
5. 预占订单仍然不是主结果，只是辅助过程信息。

---

## 6. 各区块规则明细

### 6.1 页面标题区

#### 6.1.1 作用
用于承接页面标题与返回操作。

#### 6.1.2 规则
1. 标题统一使用“产品申请工单”；
2. 当前阶段导航栏右上角不单独展示状态字；
3. 如后续需要恢复顶部状态标签，必须三种状态统一设计后再纳入。

---

### 6.2 驳回信息区

#### 6.2.1 展示条件
仅 **已驳回** 场景展示。

#### 6.2.2 应展示字段
建议包括：

- 驳回原因
- 驳回时间
- 驳回人
- 驳回节点
- 驳回备注（如有）

#### 6.2.3 规则
1. 驳回信息区需在首屏前部展示；
2. 驳回原因为主信息；
3. 驳回时间、驳回人、驳回节点为辅助信息；
4. 驳回备注应贴近驳回信息展示；
5. 驳回信息不得拆散到多个远距离区块中。

---

### 6.3 当前审批进度区

#### 6.3.1 展示条件
仅 **处理中** 且存在当前审批节点时展示。

#### 6.3.2 应展示字段
建议包括：

- 当前待处理节点
- 当前待处理人 / 待审批角色
- 当前节点状态说明

#### 6.3.3 规则
1. 当前审批进度区仅用于处理中；
2. 已驳回、已结束场景不展示此区块；
3. 该区块应服务“快速识别当前卡在哪一步”。

---

### 6.4 基础信息区

#### 6.4.1 应展示字段
建议包括：

- 工单号
- 当前状态
- 申请人
- 申请组织
- 创建时间

#### 6.4.2 规则
1. 基础信息区优先承接“识别工单身份”的字段；
2. 不重复展示已在更高优先级区块中强化的字段；
3. 不保留无信息量的冗余字段，如“工单类型：产品申请”。

---

### 6.5 发起信息区

#### 6.5.1 应展示字段
建议包括：

- 申请类型
- 申请理由

#### 6.5.2 规则
1. 发起信息区用于回答“为什么发起”；
2. 不与基础信息区混排；
3. 申请理由较长时应保证可读性。

---

### 6.6 预算与申请信息区

#### 6.6.1 应展示字段
建议包括：

- 预算号
- 预算归属组织
- 预算使用范围
- 预算总额（净额）
- 本次申请额度（净额）

#### 6.6.2 金额规则
1. 所有金额统一采用：
   - 千分符
   - 两位小数
2. “本次申请额度（净额）”为主信息；
3. “预算总额（净额）”为辅助信息；
4. 详情页不以“剩余额度（净额）”作为主金额信息；
5. 不得回退为：
   - `可用金额`
   - `总额`
   - 无千分符
   - 位数不统一

---

### 6.7 产品申请订单明细区

#### 6.7.1 区块目标
用于承接工单内多个产品申请订单明细及其产品明细。

#### 6.7.2 汇总区
建议至少展示：

- 产品申请订单明细数量
- 涉及专卖店数量
- 款数
- 件数

> 说明：  
> “款”表示 SKU 种类数；  
> “件”表示数量合计。

#### 6.7.3 卡片头
每个产品申请订单明细卡片头建议至少展示：

- 明细编号
- 订单归属专卖店
- 款数 / 件数
- 分组金额

#### 6.7.4 展开区
建议展示：

- 产品编号
- 产品名称
- 数量
- 单价 / JDE 价格
- 产品金额
- 打折标签 / 说明（如保留）

#### 6.7.5 规则
1. 默认折叠可保留；
2. 折叠态必须提供足够摘要信息；
3. 多明细场景下卡片之间需可快速区分；
4. 与创建页“产品申请订单明细”命名保持一致；
5. 不回退为“专卖店申请分组”等旧用户侧术语。

---

### 6.8 附件区

#### 6.8.1 展示规则
建议展示：

- 文件名称
- 文件大小
- 文件类型
- 预览入口
- 下载入口

#### 6.8.2 规则
1. 文件名过长时必须保证可读性；
2. 预览 / 下载入口需清晰可识别；
3. 多附件场景下排布不可混乱；
4. 已驳回场景下附件区不得抢过驳回信息区主视线。

---

### 6.9 审批流区

#### 6.9.1 应展示字段
每个审批节点建议至少展示：

- 节点名称
- 审批人
- 审批结果
- 审批时间
- 审批意见 / 备注（如有）

#### 6.9.2 节点视觉区分规则
当前页面可按以下语义区分：

- 发起节点
- 已通过节点
- 当前待处理节点
- 已驳回节点

#### 6.9.3 规则
1. 审批意见需贴近对应节点；
2. 多节点场景需保证可读性；
3. 处理中场景审批流权重较高；
4. 已结束场景审批流区低于订单结果区；
5. 已驳回场景需能快速识别驳回节点。

---

### 6.10 订单结果区

#### 6.10.1 区块目标
用于承接审批通过后生成的正式订单结果及失败原因。

#### 6.10.2 主信息
建议至少包括：

- 对应的产品申请订单明细编号
- 订单归属专卖店
- 正式订单结果状态
- 正式订单类型
- 正式订单编号
- 失败原因
- 归并说明（如有）

#### 6.10.3 正式订单类型规则
按客户层级承接：

- 客户层级 = 6 → **产品申请表订单**
- 客户层级 = 0-4 → **内部申请表订单**

#### 6.10.4 规则
1. 正式订单结果是主信息；
2. 失败原因必须贴近对应结果；
3. 不得把失败原因统一堆在区块底部；
4. 部分成功 / 部分失败场景必须可快速区分；
5. 归并生成时，必须能看出哪些明细归并到了同一正式订单。

#### 6.10.5 订单生成失败原因类型
订单生成失败时，`failReasons` 应至少覆盖以下类型：

1. **产品库存不足**：订单生成时对应产品库存不足，无法生成订单；
2. **专卖店下单标识关闭**：专卖店编号在系统中启用下单标识为关闭状态，不允许生成对应订单；
3. **订单审核被驳回到草稿状态**：订单已生成但审核环节被驳回，回退为草稿状态，需重新发起。

> 说明：  
> 以上失败原因由业务系统返回，存入 `failReasons[]`，在订单结果区贴近对应结果展示。

#### 6.10.6 专卖店变更显示规则
特殊情况下，工单通过时记录的订单所属专卖店是 A，但因 A 有异常（如库存不足），最终由 B 专卖店承接该笔订单，导致：

- 工单明细（`GroupResult.storeCode`）= A
- 正式订单实际所属专卖店（`RelatedOrder.storeCode`）= B

此时订单结果区应满足：

1. 当 `RelatedOrder.storeCode` 与 `GroupResult.storeCode` 不一致时，展示**订单实际所属专卖店信息**（`storeCode` + `storeName`）；
2. 实际专卖店信息展示位置：订单编号下方，用提示色（如橙色）区分；
3. 原始工单明细专卖店信息仍保留在卡片头，不因变更而消失。

#### 6.10.7 订单备注显示规则
以下情况需在 `RelatedOrder.remark` 中写入说明，并在订单结果区展示：

1. **多次生成订单**：备注说明重试次数、每次失败原因、最终成功条件；
2. **更换专卖店**：备注说明更换原因（如"原专卖店 A 库存不足，由专卖店 B 承接"）；
3. **其他业务特殊说明**：如订单审核被驳回后重新生成等。

备注展示位置：订单编号与实际专卖店信息之后，用辅助色（灰色、斜体）展示。

#### 6.10.8 订单结果状态固定口径（CR-20260707-002）
本工单场景下，订单结果区**只承接以下 4 种订单状态**：

| 状态 | 含义 |
|---|---|
| **已创建** | 订单已从草稿提交成为正式订单 |
| **客服已审核** | 订单已通过客服部门审核 |
| **财务已审核** | 订单已通过财务部门审核（终审） |
| **草稿** | 订单未成功提交为正式订单，或已提交后被驳回回退到草稿 |

**执行要求**：
1. 本工单详情页订单结果区**只考虑上述 4 种状态**；
2. 其他订单状态在当前工单场景中不展示、不扩展；
3. 不再使用以下表述作为正式订单状态文案：已生成、生成成功、订单已生成；
4. 如页面中存在"订单创建历史"等过程语义表述，可保留其"尝试过程"含义，但最终订单结果状态展示必须以系统正式状态为准。

#### 6.10.9 订单结果链路承接规则（CR-20260707-002）
工单审批通过后，订单结果区承接的系统链路规则如下：

1. 工单审批通过后，系统先取消关联的预占库存订单；
2. 然后调用系统现有接口，生成对应的订单草稿（产品申请表订单草稿 / 内部申请表订单草稿）；
3. 草稿生成后，系统再自动提交草稿，尝试将其提交为正式订单；
4. 自动提交成功后，订单状态承接为 **已创建**；
5. 自动提交过程中如发生校验问题，则订单回退为 **草稿**；
6. 已创建订单在后续客服审核或财务审核过程中如被驳回：原订单编号被系统物理删除，重新关联回原草稿 ID，状态回退为 **草稿**；
7. 回退为草稿后，由业务同事根据实际情况处理（编辑后重新发起 / 删除草稿 / 新建草稿并关联工单号）。

**展示承接要求**："草稿"并不等同于"未生成任何记录"，而可能表示草稿已生成但自动提交失败，或正式订单审核驳回后删除原订单编号并回退草稿。

#### 6.10.10 重试历史展示规则（CR-20260707-002）
当某关联订单存在多次尝试记录时：

1. **默认折叠**：外层仅显示"订单创建历史（N次）"入口；
2. **外层卡片只承接最新结果摘要**：
   - 最终已创建时，外层状态显示"已创建"，不得继续显示失败主提示；
   - 一直失败时，外层仅展示最新失败摘要，历史失败原因放折叠区；
   - 业务放弃时，外层只展示最新失败记录对应备注说明（灰色斜体备注样式），不展示"当前为草稿状态"等警告标题；
3. **审核驳回回草稿**：外层状态按"草稿"承接，只展示备注说明（灰色斜体样式），不额外展示警告标题；说明语义体现为"审核驳回后回草稿"；
4. **展开区内容**：可查看每次尝试的时间、订单编号/草稿编号、状态、失败原因或备注。

#### 6.10.11 重试历史排版建议（CR-20260707-002）
展开区中的失败原因/备注，建议采用以下轻量排版：时间独立一行；订单号 + 状态一行；失败原因 / 备注独立换行；避免全部挤压在最右窄列。

## 7. 驳回与重新发起规则

### 7.1 展示条件
重新发起入口仅在**已驳回**且存在重提条件时展示。

### 7.2 场景分类

#### 7.2.1 冻结期原单重提
- 原预算号沿用
- 当前场景下不允许重新选择预算号
- 创建页需承接同口径提示

#### 7.2.2 原预算已到期
- 不可继续基于原单重提
- 页面需明确提示原因
- 不得误导用户继续按原预算号发起

#### 7.2.3 非冻结期普通新建
- 可重新发起
- 按普通新建发起
- 用户自行选择预算号

### 7.3 页面表达约束
统一使用以下表达：

- 原预算号沿用
- 当前场景下不允许重新选择预算号
- 预算号不可切换 / 可切换

不得使用：

- 预算锁定
- 锁定状态
- 预算被锁定

### 7.4 数据规则
已驳回场景下，如页面需支持重新发起判断，应提供：

- `rejectionInfo`
- `reapplyCondition`

其中 `reapplyCondition` 应能支撑页面判断：

- 是否展示入口
- 是否可点击
- 按钮文案
- 提示文案
- 是否允许切换预算号

## 8. 订单结果与预占订单展示规则

### 8.1 正式订单结果生成规则
1. 审批通过后才生成正式订单；
2. 审批未通过前不得出现正式订单结果；
3. 已结束场景才是正式订单结果的主承接状态。

### 8.2 预占库存订单规则
1. 工单发起后，系统生成预占库存订单用于占用库存；
2. 审批通过后，预占库存订单取消；
3. 工单驳回后，预占库存订单同样取消；
4. 因此预占库存订单最终结果恒为“取消”。

### 8.3 页面展示规则
基于上述业务规则：

1. 预占库存订单属于**过程信息**；
2. 预占库存订单**只展示订单编号**；
3. 不展示复杂状态流转；
4. 不展示取消过程；
5. 不作为主结果区块放大展示；
6. 一般按**每笔产品申请订单明细对应一个预占订单编号**承接；
7. 页面中应明确区分：
   - 预占订单编号
   - 正式订单编号

### 8.4 分状态展示规则

#### 8.4.1 处理中
1. 处理中如需展示预占订单编号，仅在**审批流区的发起工单节点**承接；
2. 不单独新增“预占订单信息区”；
3. 不在订单结果区重复展示预占订单编号；
4. 不得让用户误解为正式订单已生成。

#### 8.4.2 已驳回
1. 已驳回场景下预占订单不是重点信息；
2. 如需保留，仅作为历史过程信息弱化展示；
3. 不单独放大为结果区主信息。

#### 8.4.3 已结束
1. 已结束场景可在结果卡片中辅助展示预占订单编号；
2. 预占订单编号依然不是主结果；
3. 正式订单结果始终高于预占订单编号。

### 8.5 强规则
**未审批通过前，详情页不得展示正式订单结果。**

因此，**处理中场景不得出现**：

- 产品申请表订单编号
- 内部申请表订单编号
- 正式订单结果成功 / 失败卡片
- 归并结果
- `retryHistory`
- 正式订单失败原因

3. 已驳回（已到期）
4. 已驳回（非冻结期）
5. 已结束（部分成功、部分失败）
6. 已结束（全部成功）
7. 已结束（归并生成）
8. 已结束（多次失败重试后成功 / 业务放弃）
9. 已结束（专卖店变更）

### 11.2 各场景规则

#### 11.2.1 处理中（多审批节点）
必须体现：
- 当前审批进度
- 多级审批流
- 多笔产品申请订单明细
- 发起节点预占订单编号
- **不得有正式订单结果**

#### 11.2.2 已驳回（冻结期）
必须体现：
- 驳回信息
- `reapplyCondition`
- 原预算号沿用
- 当前场景下不允许重新选择预算号

#### 11.2.3 已驳回（已到期）
必须体现：
- 驳回信息
- 原预算已到期
- 不可继续基于原单重提

#### 11.2.4 已驳回（非冻结期）
必须体现：
- 驳回信息
- 可重新发起
- 按普通新建承接

#### 11.2.5 已结束（部分成功、部分失败）
必须体现：
- 正式订单结果
- 失败原因贴近结果
- 成功 / 失败区分明显

#### 11.2.6 已结束（全部成功）
必须体现：
- 两种订单类型覆盖
- 无失败原因主展示

#### 11.2.7 已结束（归并生成）
必须体现：
- 多笔明细归并到同一正式订单
- 归并说明清楚

#### 11.2.8 已结束（多次失败重试）
必须体现：
- `retryHistory`
- 最终成功与业务放弃两类结果
- 重试历史默认可折叠

#### 11.2.9 已结束（专卖店变更）
必须体现：

- `RelatedOrder.storeCode` 与 `GroupResult.storeCode` 不一致
- 订单结果区展示**实际专卖店信息**（`storeCode` + `storeName`）
- `RelatedOrder.remark` 说明变更原因
- 原始工单明细专卖店信息仍保留在卡片头

### 11.3 当前禁止的 mock 错误示例

#### 错误示例 1：处理中场景出现正式订单结果
这是错误的。  
原因：

1. 正式订单仅在审批通过后生成；
2. 处理中表示仍在审批链路内；
3. 因此处理中场景不得出现订单结果卡片、正式订单编号、归并结果、重试历史。

#### 错误示例 2：预占订单被当作正式订单展示
这是错误的。  
原因：

1. 预占订单属于库存占用过程信息；
2. 其最终结果恒为取消；
3. 页面中只能弱化展示编号，不得当作最终结果主承接。

#### 错误示例 3：驳回场景缺少 `rejectionInfo` 或 `reapplyCondition`
这是错误的。  
原因：

1. 驳回详情页的核心就是解释“为什么被驳回、接下来怎么办”；
2. 缺失这两个结构会导致页面无法完成首屏承接。

#### 错误示例 4：右上角状态字只在部分状态出现
这是错误的。  
原因：

1. 处理中无、已驳回/已结束有，会造成状态承接不一致；
2. 且与基础信息区“当前状态”重复；
3. 当前阶段应统一取消导航栏右上角状态字。

#### 错误示例 5：处理中场景在多个区块重复展示预占订单编号
这是错误的。  
原因：

1. 发起节点中展示一次已足够；
2. 重复展示会增加噪音；
3. 易让用户误解为正式结果已生成。

#### 错误示例 6：最终已创建时外层仍显示失败主提示（CR-20260707-002）
这是错误的。  
原因：

1. 外层卡片只承接最新结果摘要；
2. 最终状态为"已创建"时，历史失败信息应仅在折叠区保留；
3. 外层继续显示"订单生成失败"会让用户误解当前结果仍为失败。

#### 错误示例 7：使用"已生成/生成成功"作为正式状态文案（CR-20260707-002）
这是错误的。  
原因：

1. 本工单场景下订单结果状态统一为"已创建/客服已审核/财务已审核/草稿"；
2. "已生成"是历史阶段性表述，不是系统正式订单状态；
3. 统一使用"已创建"表达"订单已从草稿提交为正式订单"。

#### 错误示例 8：业务放弃时外层堆叠多条失败备注（CR-20260707-002）
这是错误的。  
原因：

1. 业务放弃场景下，外层只应展示最新失败记录对应的业务备注说明；
2. 历史失败详情应放入折叠区；
3. 外层堆叠多条失败备注会造成信息主次混乱。

## 14. 页面容器与滚动承接说明

> 说明：  
> 本章节为 CR-20260707-003 新增，用于正式收口 Vue 参考实现的页面容器与滚动承接规则。  
> 避免后续新页面或 AI 继续接手时重复出现同类问题。

### 14.1 问题背景

在 `CR-20260707-002` 实施与验收过程中，已暴露以下问题：

1. **flex column + 100vh 布局下，`.page-content` 才是真正滚动容器**，而非 `window`；
2. **`router.scrollBehavior` 只影响 `window` 滚动**，对 `.page-content` 无效；
3. **列表 → 详情、详情 → 列表、路由切换时**如不显式复位 `.page-content`，会出现：
   - 继承上一个页面的滚动位置
   - 页面切换后左移 / 双重滚动条
   - 首屏位置不正确

### 14.2 实际滚动容器

Vue 参考实现中，移动端页面实际滚动容器为：

- **`.page-content`**（`main` 元素）

其关键样式属性：

```css
.page-content {
  flex: 1;              /* 占据 .page-container 剩余空间 */
  min-height: 0;        /* 允许 flex 子项在垂直方向收缩 */
  overflow-y: auto;     /* 内容溢出时显示滚动条 */
  -webkit-overflow-scrolling: touch;  /* iOS 平滑滚动 */
}
```

### 14.3 为什么 `scrollBehavior` 不足以覆盖

`router/index.ts` 中配置的 `scrollBehavior: () => ({ top: 0 })` 只能控制 `window` 对象的滚动位置。

但在当前 flex 布局下：
- `.page-container` 设置了 `height: 100vh; overflow-x: hidden`
- `body` 不会出现滚动条
- 实际滚动发生在 `.page-content` 内部

因此 `scrollBehavior` 对 `.page-content` 的滚动位置没有任何影响。

### 14.4 当前修正方案

在 `App.vue` 中通过 `watch(route.path)` 显式复位 `.page-content` 的滚动位置：

```typescript
watch(() => route.path, () => {
  document.querySelector('.page-content')?.scrollTo(0, 0)
})
```

**方案说明**：
1. 监听 `route.path` 变化，覆盖所有路由切换场景
2. 通过 `document.querySelector('.page-content')` 获取实际滚动容器
3. 使用可选链 `?.scrollTo(0, 0)`，容器不存在时不报错
4. 不引入新的副作用

### 14.5 适用边界

本方案适用于以下场景：

| 切换方向 | 效果 |
|---|---|
| 列表 → 详情 | 详情页从顶部开始展示 |
| 详情 → 列表 | 列表页从顶部开始展示（列表自身应独立维护滚动状态） |
| 常规页面切换 | 目标页面默认回到顶部 |
| 底部 Tab 切换 | 各 Tab 页从顶部开始 |

### 14.6 禁止事项

1. **不允许同时保留两个主滚动容器**造成双重滚动条
2. **不允许因容器布局设置不一致**导致页面切换左右偏移、首屏位置异常
3. **不允许在 `.page-content` 外层再叠加第二个主滚动层**
4. **不允许依赖 `scrollBehavior` 单独解决滚动复位问题**

### 14.7 布局层级说明

```
.page-container          (height: 100vh; flex-direction: column)
├── .app-header          (sticky; 不参与滚动)
├── .page-content   ◄──  (flex:1; overflow-y:auto) ← 唯一滚动容器
│   └── <router-view />  (页面内容)
└── .bottom-tab          (fixed; 不参与滚动)
```
