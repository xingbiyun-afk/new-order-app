# 产品申请工单“我的”页规则明细说明（v0.1）

## 1. 文档说明

### 1.1 文档名称
产品申请工单“我的”页规则明细说明

### 1.2 文档定位
本文档用于补充《产品申请工单和产品申请预算核销PRD》《产品申请工单字段清单初稿》与现有 Vue 参考实现中，关于“我的”页的页面承接规则说明。

本文档重点用于明确：

- “我的”页在当前阶段的承接边界；
- 页面哪些部分沿用 App 现有工单通用架构；
- 当前需要重点细化的是哪些内容；
- 产品申请工单在“我的”页列表中的卡片展示口径；
- Mock 数据在“我的”页中的承接原则；
- 后续 003 变更清单与 AI 开发的参考依据。

### 1.3 当前阶段定位
当前阶段，“我的”页不重新设计页面整体骨架，而是在**复用 App 现有工单页结构**的前提下，细化产品申请工单在列表卡片内容与展示数据上的承接规则。

因此，本文档不是重新定义一个全新的“我的”页，而是用于明确：

- 哪些区域跟随 App 现有架构；
- 哪些内容需要针对产品申请工单单独明确；
- 哪些 mock 数据需要补充，以支撑页面验收与走查。

---

## 2. 页面整体承接边界

### 2.1 整体结构原则
“我的”页整体结构沿用 App 现有工单页面架构，不在当前阶段重新设计页面骨架。

当前默认沿用的区域包括：

1. 页面顶部标题区
2. 搜索输入区
3. 状态 Tab 区
4. 查询筛选区
5. 工单列表区
6. 底部 TabBar

### 2.2 本期不作为重点变更的区域
以下内容本期不作为产品申请工单规则细化的重点对象：

- 顶部标题区的布局与样式
- 搜索输入框的通用交互形态
- 查询筛选区的位置、结构与基础样式
- 底部 TabBar 的结构与交互
- 整体页面骨架与通用容器布局

上述内容统一遵循 App 现有工单页架构和通用交互规范，不在本文档中重新设计。

### 2.3 本期重点细化内容
本期“我的”页需重点明确的内容为：

1. 产品申请工单列表卡片展示哪些字段；
2. 卡片中哪些字段必须展示，哪些字段明确不展示；
3. Tab 页签与当前业务范围的关系；
4. 产品申请工单 Mock 数据在列表页中的展示原则；
5. 页面验收时需要具备的基础数据覆盖效果。

## 3. “我的”页范围说明

### 3.1 当前页签范围
当前“我的”页至少承接以下页签：

- 待办
- 已办
- 已发起

### 3.2 抄送我的
当前口径如下：

- `抄送我的` 本期**不纳入正式业务承接范围**
- 但在当前 Vue 参考实现中，**保留 Tab 视觉占位**
- 本期不要求围绕“抄送我的”补完整业务规则闭环
- 后续是否正式纳入，另行通过变更清单明确

### 3.3 当前页面关注重点
当前“我的”页页面规则关注重点不在页签结构调整，而在：

- 不同页签下产品申请工单卡片如何展示；
- 当前参考实现是否有足够数量和分布合理的 mock 数据支撑走查；
- 页面是否能直观看出不同状态、不同字段内容和不同文本长度下的展示效果。

## 4. 列表卡片展示规则

### 4.1 卡片展示目标
“我的”页中的产品申请工单卡片，应以**移动端最小必要展示原则**承接，不追求一次性展示全部业务字段，而是以便于快速浏览、区分和进入详情页为目标。

### 4.2 卡片正式展示字段
当前产品申请工单列表卡片正式展示字段如下：

1. 工单类型：`产品申请`
2. 工单主状态
3. 申请人
4. 申请类型
5. 申请理由
6. 发起时间

### 4.3 卡片明确不展示字段
当前列表卡片中明确不展示以下字段：

- 工单编号
- 预算号
- 订单归属专卖店
- 客户编号
- 客户名称
- 商品明细
- 金额合计
- 附件信息
- 审批备注

### 4.4 展示原因说明
当前卡片字段收敛原则如下：

- 工单编号不在列表卡片展示，避免列表信息过重；
- 预算号不在列表卡片展示，已按最新口径移除；
- 订单归属专卖店不在列表卡片展示，避免多分组结构下信息过长；
- 多专卖店申请分组、多商品明细等复杂信息，统一在详情页承接；
- 列表页重点用于快速浏览和进入详情，而非完整承载工单全部业务结构。

## 5. 字段展示规则细化

### 5.1 工单类型
- 固定展示：`产品申请`
- 用于与其他工单类型区分

### 5.2 工单主状态
当前正式展示口径为：

- 处理中
- 已结束
- 已驳回

说明：
- 列表卡片展示的是主状态映射口径；
- 后端实际状态码以后续正式接口为准；
- 当前参考实现按业务展示语义承接。

### 5.3 申请人
- 展示发起人姓名
- 用于快速识别工单来源

### 5.4 申请类型
- 来源于预算配置带出的申请类型
- 例如：新品申请、活动申请、补货申请等
- 卡片中作为主体信息展示

### 5.5 申请理由
- 来源于预算配置带出的申请理由
- 在卡片中允许做截断展示
- 长文本不要求在列表页完整展开

### 5.6 发起时间
- 列表卡片中展示字段为：发起时间
- 格式为：

`月（00）-日（00） 时（00）-分（00）`

例如：
- `06-26 10:17`
- `06-25 14:30`

### 5.7 排序说明
排序逻辑沿用现有工单规则：

- 最新到达优先
- 最久等待优先

说明：
- 列表卡片显示的是发起时间；
- 但排序逻辑不要求严格按发起时间排序；
- 仍沿用现有工单“到达顺序 / 等待时长”逻辑承接。

## 6. 搜索、筛选与结构边界说明

### 6.1 搜索区
搜索区沿用 App 现有工单通用搜索承接方式，本期不对搜索区交互范式做重新设计。

当前阶段本文档不重点展开：

- 搜索区样式规则
- 搜索框位置规则
- 搜索图标与占位样式规则

上述内容统一跟随 App 现有工单架构。

### 6.2 筛选区
筛选区同样沿用 App 现有工单页通用承接方式，本期不重新定义筛选结构。

例如：
- 全部工单
- 排序入口

其位置、样式、交互形式跟随现有工单通用实现。

### 6.3 底部 TabBar
底部 TabBar 属于 App 通用导航结构的一部分，本期不在产品申请工单范围内单独改造。

说明：
- 发起工单
- 我的
- 组内

等底部导航结构，均按 App 现有架构沿用。

## 7. Mock 数据承接规则

### 7.1 当前 Mock 数据定位
“我的”页中的 mock 数据，当前主要用于：

1. 支撑页面结构验收；
2. 支撑卡片字段展示走查；
3. 支撑不同状态、不同文本长度、不同时间展示效果验证；
4. 为后续继续交给 AI 开发或调整样式提供可视化基础。

### 7.2 Mock 数据重点不是“有数据即可”
当前“我的”页的 mock 数据补充，应重点满足以下目标：

- 能看到足够数量的列表卡片；
- 能覆盖不同主状态；
- 能覆盖不同长度的申请理由；
- 能覆盖不同时间展示效果；
- 能支撑待办 / 已办 / 已发起的基本验收；
- 若保留“抄送我的”Tab 视觉占位，也需明确该 Tab 当前是否为空态或使用占位数据。

### 7.3 当前建议的数据覆盖思路
后续补 mock 数据时，建议至少考虑：

- 短申请理由
- 长申请理由（用于截断展示）
- 处理中状态卡片
- 已结束状态卡片
- 已驳回状态卡片
- 不同申请类型分布
- 不同发起时间分布

### 7.4 当前不在本文档中写死的数据条数
当前文档先不固定每个 Tab 必须展示多少条 mock 数据。  
具体 mock 数据数量与分布，可在后续开发验收基础上，通过 003 变更清单进一步明确。

原因如下：

- 当前需先完成 002 变更项开发与验收；
- 003 将更适合统一承接“我的”页展示补强与 mock 数据补充；
- 数据条数应结合页面实际观感、滚动效果和验收需求再定。

## 8. 当前页面验收关注点

“我的”页在当前阶段的验收重点包括：

1. 页面整体结构是否与 App 现有工单架构保持一致；
2. 搜索区、筛选区、底部 TabBar 是否保持通用样式与位置；
3. 产品申请工单卡片字段是否与现行规则一致；
4. 是否已去掉不应出现在卡片中的字段（如预算号）；
5. 不同状态下卡片视觉区分是否清晰；
6. 长申请理由是否能合理截断；
7. 发起时间格式是否符合当前规则；
8. 不同页签下是否有足够数据支撑页面走查；
9. “抄送我的”是否仅作为视觉占位而未误承接正式业务。
