## 二、本轮实现

### 2.1 §8 依赖安全：xlsx → exceljs

| 项 | 变更前 | 变更后 |
|---|---|---|
| 依赖 | `xlsx@0.18.5` | `exceljs@4.4.0` |
| npm audit | 1 high severity（`xlsx`） | 2 moderate severity（`exceljs` 传递 `uuid`） |
| API 风格 | 同步 `XLSX.read/sheet_to_json` | 异步 `new ExcelJS.Workbook().xlsx.load()` + `eachRow` |

**降级收益**：`npm audit` 风险从 **1 个 high severity（`xlsx`）** 降为 **2 个 moderate severity（`uuid` 传递依赖）**。

**当前遗留风险判断**：
- 当前剩余告警为 `uuid < 11.1.1` 的 **moderate** 漏洞；
- 来源于 `exceljs@4.4.0` 的**传递依赖**，并非当前业务代码直接依赖；
- 结合本轮实现范围，当前主要影响为**依赖安全告警层面**，未识别出本轮产品申请工单发起 / 导入链路中的直接可利用入口。

**本轮未继续强制修复原因**：
- 若执行 `npm audit fix --force`，会将 `exceljs` **强制降级至 `3.4.0`**；
- 该调整属于 **breaking change**，可能影响本轮刚完成的：
  - Excel 文件导入解析；
  - `exceljs` 动态加载；
  - 模板读取与导入兼容性；
  - 相关 mock / 前端承接逻辑；
- 综合判断回归风险高，因此本轮**不采用**强制自动修复方案。

**后续处理建议**：
1. 优先跟踪 `exceljs` 上游对 `uuid` 依赖的升级修复；
2. 若后续安全要求提升或仓库进入正式生产交付阶段，再专项评估：
   - 平滑升级 `exceljs`；
   - 或替换为其他 Excel 解析库；
3. 在后续开发总结 / 风险台账中持续保留该项遗留说明，避免误判为“已完全清零”。

### 2.2 §7 动态加载 exceljs

| 项 | 变更前 | 变更后 |
|---|---|---|
| 主包体积 | 509 kB（gzip 170 kB） | **180 kB**（gzip 58 kB） |
| 副包（异步） | — | `exceljs.min-*.js` 930 kB（gzip 256 kB），按需加载 |

实现：`mocks/index.ts` 顶部 `import ExcelJS` 改为 `parseFileImport` 内部 `await import('exceljs')`。

**结论**：批量导入能力仅在用户点击"上传文件"时才下载，主包瘦身 65%。

### 2.5 §3 产品选择页

`ProductSelectView.vue` 完整重写（50 行 → 220+ 行）：
- **去 `>`**：卡片右侧移除 chevron，避免"可继续进入下一级"误判。
- **折扣字段标签化**：原"折扣：5.0折"独立行改为"打折 50%" 标签，与 #FF9800 警告色系。
- **卡片风格统一**：与 BudgetSelectView / StoreSelectView 同一视觉规范。
- **千分符**：已在 §6 完成。
- **打折属性区分（PRD 依据）**：
  - 打折产品（`isDiscount: true`）：显示橙底"打折 50%"标签。
  - 不打折产品（`isDiscount: false`）：不展示折扣标签。
  - 依据：PRD §14.3「产品选择页可保留折扣辅助展示」+ §9.8-4「页面不单独展示折扣字段，仅展示'是否打折产品：是/否'」。选择页仅打折产品展示折扣。
  - mock 数据：`isDiscount: true` 时 `discount: 0.5`（5.0 折），`isDiscount: false` 时 `discount: 1`（原价×1，业务口径）。
  - 金额计算：`calculateAmount()` 按 `isDiscount` 分支 — 打折 = `JDE价格 × 数量 × 折扣`，不打折 = `JDE价格 × 数量`。

- **明细头三元素 24px 行高统一（review fix）**：明细头右侧"数字胶囊 / 删除按钮 / 折叠 icon"三者统一 `height: 24px; line-height: 1;` + inline-flex 居中容器，保证水平基线对齐。

### 2.8 §2 驳回后重新发起承接

实现方案：冻结期完整实现 + 非冻结期方案 B（按普通新建）。

**核心改动**：
- `mocks/index.ts` 新增 `findBudgetByWorkOrderNo(workOrderNo)`：演示用映射 `PA202406260001` → `mockBudgets[4]`（id='5'，freezeStartDate=2026-07-01，expiryDate=2026-08-10，TODAY 处于冻结期）。
- `CreateView.vue onMounted`：识别 `?rejectedFrom=PA202406260001` 路由参数 →
  - 查 `findBudgetByWorkOrderNo` 找到原预算
  - **冻结期**（`freezeStartDate <= TODAY < expiryDate`）：`store.selectBudget(originalBudget.id)` 带入且不允许切换预算号；`goBudgetSelect` 已被拦截不可切换；UI 已有 #FFF3E0 提示。
  - **已到期**：`selectBudget` 让用户看到"已到期"红色提示，不可提交。
  - **非冻结期**（方案 B）：不携带原预算，selectedBudget 保持 null，按普通新建处理，用户自行选预算。

**未实现部分**：
- 详情页"驳回后重新发起"按钮：本轮只做承接端，详情页按钮展示条件/可见性按变更清单 §5.5 沉淀到"详情页规则文档"，本轮不展开。
- 入口条件判断（哪些状态可重新发起）：同上。
- 后端接口：`findBudgetByWorkOrderNo` 是 mock 实现，真实环境替换为后端请求。
