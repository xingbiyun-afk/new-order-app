### 5.2 体积来源初步分析

变更清单 §12.2 假设 `xlsx` 相关能力 + `SkuBatchImportView` 是主要怀疑来源。

**本轮实地核对结果**：
- **当前仓库中不存在 `SkuBatchImportView.vue` 文件**。`vue-reference/src/views/work-order-budget/` 实际包含：`BudgetSelectView.vue`、`DetailView.vue`、`StoreSelectView.vue`、`CreateView.vue`、`ProductSelectView.vue`。
- `xlsx` 包的**真实使用点**为：
  1. `vue-reference/src/mocks/index.ts` 行 1（`import * as XLSX from 'xlsx'`）+ 行 291（解析 `.xlsx`/`.xls` 文件名）
  2. `vue-reference/src/views/work-order-budget/CreateView.vue` 行 565 / 568（文件导入提示文案 + `<input type="file" accept=".xlsx,.xls,.csv">`）

### 5.3 动态加载改造方案评估（待后续轮次实施）

| 方案 | 改动文件 | 风险 | 收益 |
|------|----------|------|------|
| A. `mocks/index.ts` 整体 async 化 | 1 | 中（mocks 是 mock 数据，async 化改动大） | 主包瘦，但可能引发连锁改动 |
| B. 仅 `xlsx` import 改为 `await import('xlsx')` | 1-2 | 低 | 仅在用户点击"上传文件"时才加载，行为接近按需 |
| C. 暂不动 | 0 | 无 | 留待打包优化专项 |

**建议方案 B**（风险最低、最贴近变更清单 §12.3 第 2 条方向）。本轮**不实施**，留待代码阶段处理。

### 5.4 依赖安全评估

```text
# npm audit report

xlsx  *
Severity: high
Prototype Pollution in sheetJS - https://github.com/advisories/GHSA-4r6h-8v6p-xvw6
SheetJS Regular Expression Denial of Service (ReDoS) - https://github.com/advisories/GHSA-5pgg-2g8v-p4x9
No fix available
node_modules/xlsx

1 high severity vulnerability
```

**关键结论**：
- 漏洞来源包：`xlsx`（直接依赖，`package.json` 中 `^0.18.5`）
- 漏洞类型：Prototype Pollution + ReDoS
- 官方修复状态：**No fix available**（xlsx 维护基本停滞）
- 修复选项：
  1. 升级到 `xlsx` 的活跃 fork（如 `xlsx-js-style`、`@sheet/core`），兼容性风险大
  2. 替换为 `exceljs`（无已知漏洞，API 类似，但体积接近）
  3. 替换为 `read-excel-file`（轻量，仅支持读取不支持写入）
  4. 接受风险：当前为 Vue 参考实现 / Mock 阶段，不连生产，参考实现用途下风险可控

**本轮选择**：仅记录评估结论，**不动** `package.json` / `package-lock.json`。最终修复方式在代码阶段与用户确认后处理。
