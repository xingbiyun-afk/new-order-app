# CR-20260703-003 开发总结：产品申请工单详情页驳回态与重新发起链路优化

## 1. 变更单号

`CR-20260703-003`

## 2. 变更摘要

本轮变更以"详情页驳回态与重新发起链路"作为主线，完成了以下工作：

### 2.1 类型定义扩展（`src/types/index.ts`）

新增 3 个类型：
- `RejectionInfo`：驳回信息（驳回时间、驳回人、驳回节点、驳回原因、驳回备注）
- `ReapplyAvailability`：重新发起条件枚举（`available` / `budget_expired` / `freeze_period_locked` / `non_freeze_new`）
- `ReapplyCondition`：重新发起条件判断结果（可用性、提示信息、原预算号、是否允许切换预算）

扩展 `ProductWorkOrder` 接口，新增 `rejectionInfo` 和 `reapplyCondition` 两个可选字段。

### 2.2 Mock 数据补强（`src/mocks/index.ts`）

新增 5 组场景数据 + 1 个场景分发函数：

| 场景 | 变量名 | 状态 | 关键特征 |
|------|--------|------|----------|
| 处理中（多明细） | `mockWorkOrderProcessing` | 处理中 | 3 个分组、4 个附件、待处理审批节点 |
| 已驳回-冻结期 | `mockWorkOrderRejectedFreeze` | 已驳回 | 冻结期预算 B2024Q3005、`freeze_period_locked`、不可切换预算 |
| 已驳回-已到期 | `mockWorkOrderRejectedExpired` | 已驳回 | 原预算已到期、`budget_expired`、可切换预算 |
| 已驳回-非冻结期 | `mockWorkOrderRejectedNonFreeze` | 已驳回 | 非冻结期、`non_freeze_new`、可切换预算 |
| 已结束 | `mockWorkOrderCompleted` | 已结束 | 审批通过、部分订单成功+部分失败 |

分发函数：`getMockWorkOrderDetail(scene?)` 支持通过 `?scene=` query 参数或路由 ID 匹配场景。

### 2.3 详情页重构（`src/views/work-order-budget/DetailView.vue`）

完整重写，主要优化点：

1. **驳回信息区（首屏 Banner）**：已驳回状态在页面顶部展示醒目 banner，包含驳回原因、驳回人、驳回时间、备注，用户无需滚动即可理解"为什么被驳回"
2. **已到期不可重提提示**：原预算到期时展示独立提示条，明确告知不可继续基于原单重提
3. **重新发起入口**：已驳回状态底部固定展示重新发起按钮，根据 `reapplyCondition.availability` 自动判断按钮文案和禁用态
4. **冻结期确认弹窗**：点击"基于原单重新发起"时弹出确认弹窗，明确提示"原预算号沿用、不允许切换预算号、按剩余额度校验"
5. **金额口径统一**：全部金额字段使用 `formatAmount` 千分符格式化，文案统一为"剩余额度（净额）"、"预算总额（净额）"
6. **产品标签化**：打折产品展示"打折 N%"标签（延续 CR-20260703-001 口径）
7. **命名口径统一**："产品申请订单明细"替代"专卖店申请分组"，"产品"替代"商品"
8. **附件区增强**：展示文件大小、文件类型图标颜色区分（PDF红/Excel绿/Word蓝/Zip橙）
9. **审批流优化**：节点圆点颜色区分状态（发起青绿/通过绿/驳回红/待处理橙），关联订单展示在对应审批节点下
10. **订单结果区优化**：失败原因独立展示（红色警告卡片），订单状态左边框颜色区分
11. **金额展示精简**（用户反馈修改）：预算总额灰色弱化 + 本次申请额度主色突出，去掉剩余额度和金额合计
12. **产品明细汇总与折叠**（用户反馈修改）：新增汇总行（共X笔/Y个专卖店/Z个产品），卡片默认折叠，编号改为胶囊样式

### 2.4 路由 ID 场景映射

在 DetailView 中实现 ID→场景映射，使列表页点击进入详情页时自动匹配对应场景：
```
'2' → processing         (多明细+多附件)
'3' → rejected-freeze    (冻结期原单重提)
'4' → rejected-expired   (原预算已到期)
'5' → rejected-nonfreeze (非冻结期普通新建)
'6' → completed          (已结束+部分失败)
```

### 2.5 uuid moderate 遗留问题评估

当前状态：
- `uuid@8.3.2` 存在于 `exceljs@4.4.0` 的传递依赖链中
- `npm audit` 在镜像上不可用，但根据历史记录 moderate 漏洞仍存在
- 上游 `exceljs` 最新版本（4.4.0）仍锁定 `uuid@^8.3.0`

下一步路径建议（记录于开发总结，未在本轮执行）：
- **路径 A**：继续保留 `exceljs`，等待上游修复（推荐，当前阶段）
- **路径 B**：正式工程阶段评估替换为 `SheetJS` 或其他方案
- **路径 C**：弱化 Excel 文件导入依赖，仅保留文本/CSV 能力

## 4. 关键决策

1. **不新建详情页规则明细文档**：按变更单 §10.3 要求，本轮仅做轻量梳理和页面实现，待字段结构稳定后再正式建档
2. **不引入"预算锁定"概念**：严格按变更单 §5.4，相关场景表达为"原预算号沿用/不允许切换预算号"，不新增业务概念
3. **重新发起弹窗文案差异化**：冻结期弹窗明确提示"不允许切换预算号"，非冻结期提示"可自行选择预算号"，避免用户困惑
4. **uuid moderate 本轮不修复**：仅做路径评估和书面沉淀，不以高风险方式在本轮强制修复

## 5. 关键发现与遗留

1. **沙盒构建环境限制**：本次 npm install 受限于文件系统（tar entry ENOENT、native binary EACCES），最终通过 Vite 5 + esbuild-wasm 替代方案完成构建
2. **Tailwind CSS v4 移除**：`@tailwindcss/vite` 依赖 `@tailwindcss/oxide` native binding，沙盒环境无法执行，本轮移除（项目组件中未使用 Tailwind utility class，视觉无影响）
3. **Vue 3.5.13 包版本一致性**：构建过程中发现 `@vue/*` 包版本不一致导致运行时错误，已统一所有相关包到 3.5.13
4. **未处理**：uuid moderate 漏洞、详情页规则明细文档建档（待后续 CR）
