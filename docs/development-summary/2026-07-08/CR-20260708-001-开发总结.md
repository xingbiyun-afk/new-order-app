## 本轮目标

本轮定位为**规则补强 / 展示修正 / Mock 治理 / 文档回写**，核心目标不是新增业务能力，而是把当前已进入执行阶段的预算选择页与详情页，在以下 3 类问题上补齐到可直接验收状态：

1. 预算选择页缺少 `isAbnormal=true` 但仍可申请的正式 mock 验证样本 → **规则文档回写**（Mock 已有）；
2. 详情页订单创建历史中"删除草稿理由"与"失败原因/审核驳回说明"之间的关系，需要先在规则文档中补充正式说明，为后续页面展示收口提供依据；
3. 详情页当前 mock 子场景覆盖不全（仅 3/6），审核驳回回草稿变体缺失 → **Mock 补强**。

## 技术决策

### 1. 本轮以规则补强与 Mock 补强为主

本轮优先完成以下两类工作：

1. 预算选择页“异常但可选”样本的规则文档显式化；
2. 详情页场景 8 / 9 的子场景补强与规则文档回写。

对于“删除理由 / 备注显示 / 订单创建历史”的最终页面承接，本轮先补规则说明与样本基础，未在本轮作为最终展示收口结论。

### 2. 审核驳回回草稿子场景的文案区分

新增的审核驳回回草稿子场景中，failReason 统一以“订单审核被驳回到草稿状态”开头，与自动提交失败的 failReason（如“产品库存不足…”“专卖店下单标识关闭…”）形成文案差异，便于后续继续验证两类场景的展示优先级。

### 3. 删除理由与备注显示的最终承接口径留待下一轮收口

本轮虽已在规则文档中补充“删除理由 / 驳回回草稿说明 / 自动提交失败原因”的关系说明，但页面外层摘要、展开区历史记录、`outerRemark` 与 `retryHistory` 的最终承接模型，仍需在后续变更清单中继续细化并统一收口。
