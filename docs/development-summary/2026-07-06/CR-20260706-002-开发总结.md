## 1. 审批流区优化（变更清单 §4）

### 1.1 节点名称字段化

- `ApprovalNode` 接口新增可选字段 `nodeName?: string`
- 所有 mock 场景的审批节点补充 `nodeName`（如"一级审批""二级审批"）
- 模板优先展示 `nodeName`，兜底显示"审批节点 ${i}"

### 1.2 节点类型视觉区分

新增 `getNodeClass()` 方法，按节点类型和结果返回 CSS 类名：

| 节点类型 | CSS 类名 | 视觉效果 |
|---|---|---|
| 发起工单（start） | `node-start` | 标题青绿色高亮 |
| 已通过（result=通过） | `node-passed` | 正常显示 |
| 当前待处理（result=待处理） | `node-current` | 橙色背景高亮 + 待处理人加粗 |
| 已驳回（result=驳回） | `node-rejected` | 红色背景高亮 + 驳回意见左侧红色边框 |

### 1.3 审批意见贴近节点

- `.node-remark` 改为卡片样式（浅灰背景 + 圆角）
- 已驳回场景下，驳回节点的 `.node-remark` 加左侧红色边框，视觉上紧贴驳回节点

## 2. 订单结果区优化（变更清单 §5）

### 2.1 预占库存订单展示收口（§6）

**变更前**：审批流区和订单结果区均展示"功能订单：XXX（状态）"

**变更后**：
- 只展示预占库存订单编号，不展示状态
- 文案改为"预占订单：XXX"（而非"功能订单"）
- 字号降级为 12px、颜色 `#999`，降为辅助字段

涉及修改位置：
- 审批流区：`.node-function-order`
- 订单结果区：`.result-function-order`

### 2.2 失败原因贴近对应结果

- 失败原因已通过 `.tip-banner.tip-error.tip-inline` 紧贴对应 `result-card` 展示
- 无需额外修改（CR-20260706-001 已完成此部分）

| 必验场景 | 覆盖场景 | 说明 |

| 5. 已结束（存在归并生成） | `completed-merged` | 新增，两笔明细归并到同一正式订单 |
| 6. 每笔明细均带预占订单编号 | 全部场景 | `functionOrderNo` 全覆盖 |
| 7. 审批流较完整 | `processing`/`completed` | 多级审批节点 |
| 8. 订单结果较复杂 | `completed`/`completed-full`/`completed-merged` | 成功/失败/归并 |
| 9. 两种订单类型覆盖 | `completed`/`completed-full` | 产品申请表订单 + 内部申请表订单 |

## 4. 白功能项：`uuid` moderate 风险复核

**结论**（变更清单 §10 要求）：
1. 风险来源：`exceljs@4.4.0` 的传递依赖，非当前业务直接依赖
2. 不采用 `npm audit fix --force`：会破坏现有 Excel 导入能力
3. 影响范围：仅 `parseFileImport()` 中的 Excel 解析能力
4. 后续建议路径：
   - 关注 `exceljs` 上游修复
   - 评估 `uuid` 升级可行性
   - 考虑 SheetJS 社区版 / PapaParse 纯 CSV 方案
