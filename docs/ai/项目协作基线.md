# 项目协作基线

> 本文档为 `new-order-app` 仓库的通用协作基线文档。  
> 适用于 Kimi Agent、Codex、Trae、SOLO、ChatBox + GPT 及其他 AI 协作工具。  
> 后续所有需求梳理、文档修改、原型整理、代码开发前，默认需先阅读本文档。  
> 如本文档与历史“预算申请工单”相关文档冲突，以本文档的“产品申请工单 / 产品申请预算核销”现行口径为准。

---

## 一、文档定位

本文档不是某一次 CR 的变更记录，也不是单一页面 PRD，而是：

- `new-order-app` 仓库的项目级协作基线
- 多 AI 工具协作时的统一前置约束
- 后续 App / Web 工单相关需求继续推进前的默认口径说明
- 防止 AI 误用历史“预算申请工单 / 预算责任人 / React 正式开发”等旧口径的基准文档

后续以下工作均需默认遵守本文档：

- 产品需求梳理
- 文档修改
- 原型页面规划
- Vue 正式开发前的需求拆解
- Mock 设计
- 字段和状态说明
- 跨端 UI 复用说明
- AI 任务执行

## 二、仓库当前定位

`new-order-app` 早期以 React + TypeScript + Vite 初始化，用于沉淀现有“工单处理中心”的移动端样式、交互结构和产品申请预算核销相关原型参考。

根据当前最新项目口径：

- 当前仓库不作为正式 App 生产代码主仓；
- 当前仓库保留移动端样式基线、跨端 UI 复用策略和产品申请预算核销需求文档；
- App 端正式开发按 Vue 框架体系推进，具体版本以正式 App 工程为准；
- Web 端正式开发按 Vue2 框架推进；
- 当前 React/Vite 初始化内容仅作为移动端样式、交互和结构参考；
- 旧“预算申请工单”口径已调整为“产品申请工单 / 产品申请预算核销”。

## 三、当前最高优先级业务口径

后续所有 AI 工具必须优先遵守以下口径：

1. 当前正式业务对象为：**产品申请工单**。
2. 产品申请工单主要用于：**预算核销**。
3. 旧“预算申请工单”仅为历史口径，不作为后续正式需求命名。
4. 预算责任人体系已下线，不再使用主责、备责、责任人异常等概念。
5. 发起人和审批人规则来源于：**产品申请审批流程管理**。
6. 预算状态仅包含：草稿、已生效、已作废、已到期。
7. 不存在“待调整”状态。
8. `applicationAvailable` 是预算核销可用标识。
9. `isAbnormal` 仅表示组织异常相关影响标识，不等于预算不可核销。
10. 正式状态、审批流、订单状态以后端 / 正式 Vue 工程口径为准。
11. 不得自行定义正式工单状态、审批状态、功能订单状态、产品申请表订单状态或内部申请表订单状态。

## 六、产品申请工单 / 预算核销核心规则

### 6.1 工单入口与命名

入口名称统一为：

```text
产品申请工单
```

历史“预算申请工单”仅作为旧口径参考。

### 6.2 预算核销时点

当前已确认：

- 提交产品申请工单并进入审批流时，占用预算；
- 审批中，该金额进入申请中金额；
- 驳回后，释放申请中金额；
- 审批通过后，申请中金额转为已用金额；
- 审批通过即视为预算核销完成；
- 后续订单失败不自动回滚预算核销。
- 如后续订单因库存、产能或其他原因无法最终完成，由财务和业务部门线下协商处理；必要时由财务手动增加下一季度预算。

### 6.3 金额字段来源

正式实现以后端返回为准。

建议接口返回：

- 预算金额
- 已用金额
- 申请中金额
- 可用金额
Mock 或原型阶段可参考：

```text
可用金额 = 预算金额 - 已用金额 - 申请中金额
```

但必须在文档或代码注释中注明：

```text
正式实现以后端返回 availableAmount / applyingAmount / usedAmount 为准。
```

### 6.4 审批路径匹配规则

当前已确认审批路径匹配依据为：

1. 发起人组织身份
2. 预算归属组织
3. 预算适用范围

不得自行加入以下未确认条件：

- 金额区间
- 产品类型
- 费用类型
- 客户类型
- 其他 AI 自行推测的审批条件
- 
如后续需要扩展，必须通过新的需求确认。

### 6.5 功能订单预占库存

当前已确认功能订单仍然存在：

1. 产品申请工单提交进入审批流后，同时生成对应功能订单；
2. 功能订单用于预占产品库存，不产生交易；
3. 审批中，功能订单持续占用库存；
4. 审批通过或驳回后，功能订单取消，库存释放。

### 6.6 订单生成规则

产品申请工单审批通过后，根据客户层级生成订单：

| 客户层级 | 生成订单 |
|---|---|
| 6 | 产品申请表订单 |
| 0-4 | 内部申请表订单 |

该规则为现有系统成熟固定规则，本次需求只需写明并衔接，不重新设计订单类型判断逻辑。

### 6.7 订单生成失败处理

订单生成失败时：

1. 订单回退为草稿状态；
2. 必须展示失败原因；
3. 允许用户修改失败草稿后重新发起；
4. 允许用户从现有产品申请表订单或内部申请表订单新建草稿；
5. 新建草稿时可关联对应审批通过的产品申请工单号，以绑定预算核销信息；
6. 订单失败不代表产品申请工单审批回退；
7. 订单失败不代表预算核销自动回滚。

## 七、预算管理集成规则

预算管理规则来源于 `new-order-system` 的预算管理模块。

App / Vue 端不重做预算管理。

### 7.1 当前有效预算状态

仅允许：

- 草稿
- 已生效
- 已作废
- 已到期

不得重新引入：

- 待调整

### 7.2 可核销预算号基础条件

产品申请工单可选择的预算号至少应满足：

1. 预算状态为已生效；
2.  `applicationAvailable = 可申请` ；
3. 可用金额大于 0；
4. 发起人具备预算核销资格。

### 7.3 预算核销资格

预算核销资格由以下两个条件共同决定：

1. 预算归属组织：员工所属组织必须在预算归属组织范围内；
2. 预算适用范围：员工角色必须在预算适用范围内。

不存在独立的预算核销资格名单。

### 7.4  `isAbnormal` 使用限制

 `isAbnormal` 仅表示组织异常相关影响标识。

不表示：

- 审批人员异常
- 预算不可核销
- 正式冻结状态
- 最终业务规则

不得将其作为预算不可核销的判断条件。

## 八、跨端 UI 复用策略

本系统工单存在 App 端和 Web 端。

### 8.1 App 端

App 端是移动端工单样式的主要参考来源。

当前已有移动端工单样式包括：

- 工单处理中心
- 发起工单
- 我的
- 待办
- 已办
- 抄送我的
- 已发起
- 工单详情
- 工单流程时间线

“组内”入口当前未开通，不纳入本期功能范围。

### 8.2 Web 端

Web 端存在两类页面：

1. 工单查询类页面：使用 Web 管理后台查询表格样式；
2. 发起、待办、已办、抄送我的、已发起、详情、审批等流程页：沿用移动端工单样式。

### 8.3 复用边界

当前阶段的“复用”主要指：

- 视觉规范复用
- 交互结构复用
- 组件拆分思路复用
- 字段展示口径复用

不代表：

- 跨仓库直接复用 React 代码
- Web 端直接 import App 仓 CSS
- 正式 Vue 工程直接照搬 React 组件

## 九、文档更新约束

后续修改文档时，必须注意：

1. 标题改了，正文也要同步改；
2. 旧“预算申请工单”如保留，必须标记为历史口径；
3. 不得产生两个互相冲突的“现行说明文档”；
4. 如文档目录沿用 work-order-budget 旧命名，必须注明其历史命名原因；
5. 新增文档需更新对应 README 索引；
6. 历史内容可保留，但必须明确“历史参考 / 不作为现行依据”。

## 十、开发执行约束

1. 禁止直接推送 main；
2. 必须基于独立分支开发；
3. 不得在当前仓库中开发正式生产页面；
4. 不得基于旧“预算申请工单”口径开发新页面；
5. 不得新增正式状态枚举；
6. 不得恢复预算责任人体系；
7. 不得恢复“待调整”状态；
8. 不得自行扩展审批路径规则；
9. 不得新增未确认的 Mock 业务规则；
10. 如只是文档任务，不得修改代码逻辑。

## 十一、需求不明确时的处理规则

遇到以下情况，必须先输出问题清单，不得自行决定：

- 入口名称不确定
- 字段是否必填不确定
- 某个旧术语是否仍可使用不确定
- 审批路径匹配条件不明确
- 预算金额占用 / 释放 / 核销时点不明确
- 功能订单失败处理不明确
- 订单生成失败处理不明确
- 是否需要通知发起人 / 审批人不明确
- 是否允许一张产品申请工单拆成多张订单不明确
- 是否存在人工调整已核销金额入口不明确
- 问题清单建议格式：

| 序号	| 位置	| 当前理解 | 疑点 | 备选方案 | 建议 |

## 十二、后续 AI 执行默认要求

后续凡交给 Kimi、Codex、Trae 或其他 AI 的任务，默认必须遵守以下要求：

1. 先阅读本文档；
2. 再阅读 CLAUDE.md；
3. 如涉及移动端样式，阅读 docs/technical/mobile-ui-baseline.md；
4. 如涉及跨端复用，阅读 docs/technical/cross-end-work-order-ui-strategy.md；
5. 如涉及预算核销，阅读 docs/product/work-order-budget/product-application-budget-writeoff-rules.md；
6. 如涉及预算管理规则，阅读 docs/product/work-order-budget/budget-management-integration.md；
7. 输出需求理解时，先复述受影响的当前基线；
8. 如发现历史文档与当前基线冲突，以当前基线为准，并指出冲突点；
9. 如定义模糊，先列问题清单，不得直接开发；
10. 输出结果时，必须说明：

- 修改了哪些文件
- 是否涉及文档更新
- 是否存在遗留问题
- 是否引入新的待确认点

## 十三、最终原则

如果某项实现、文档或命名虽然“技术上能跑通”，但会破坏当前统一口径、重新引入已废弃概念，或让后续协作产生歧义，则视为不合格。

后续目标不只是“页面能运行”，而是：

代码、文档、命名、业务语义保持一致，并且可持续交给不同 AI 工具继续协作。
