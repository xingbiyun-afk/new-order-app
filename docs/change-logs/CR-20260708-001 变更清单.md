本 CR 属于**现行正式规则补强与参考实现修正**，目标不是新增业务能力，而是把当前已进入执行阶段的预算选择页与详情页，在以下 3 类问题上补齐到可直接验收状态：

1. 预算选择页缺少“`isAbnormal=true` 但仍可申请”的正式 mock 验证样本；

本 CR 执行后，应形成：

- 可直接用于 AI 开发的规则依据；
- 可直接用于页面验收的展示口径；
- 可直接用于 mock 走查的场景样本；
- 可直接回写的正式文档记录。

## 4. 变更背景

### 4.1 预算选择页异常样本缺失
当前预算选择页 mock 数据缺少一类正式业务上必须存在的预算样本：

- 预算仍在有效申请周期内；
- 剩余额度充足；
- `applicationAvailable=true`；
- 但 `isAbnormal=true`。

该类预算按照既有固定口径应当：

- **允许选择**
- **展示异常提示**
- **不阻断申请**

此前该类样本曾存在，后续在预算选择页面规则明细文档和页面优化过程中被遗漏或移除，导致当前页面无法完整验证“异常仅提示、不阻断”的正式规则。

### 4.2 订单创建历史备注承接不够精细
当前详情页订单结果区已收口“业务放弃时外层只展示最新备注”“最终成功时外层不显示失败提示”等规则，但在以下子场景上仍有语义混乱风险：

- 自动提交失败后，原草稿被业务主动删除且不再发起；
- 自动提交失败后，原草稿被删除，同时新建一份草稿并继续尝试发起；
- 已创建订单在客服/财务审核中驳回回草稿后，原草稿被删除或改为其他处理方式。

现有展示中，可能出现：

- 删除理由与旧失败原因并列主展示；
- 同一条历史记录出现两条语义不同的备注；
- 已被业务主动删除的草稿，仍以“失败原因”为主语义展示，而不是以“删除理由”为主语义展示。

这会直接影响页面主次关系与验收判断。

### 4.3 详情页 Mock 子场景承接不足
虽然 `CR-20260707-003` 已建立详情页主场景样本总表，但当前 mock 对“订单创建历史”的承接仍偏主场景级别，尚不足以让开发、测试、AI 在以下差异上做到快速识别：

- 原草稿继续编辑重试
- 删除原草稿后业务放弃
- 删除原草稿后新建草稿再发起
- 驳回回草稿后的删除与重提差异
- 删除理由与失败原因在不同子场景下的展示优先级

因此需要对规则文档、mock 注释、必要的 mock 数据组织方式进行补强。

### 8.1.2 展示规则
对于上述预算样本，页面必须满足：

1. 可正常显示预算卡片；
2. 可展示预算编号；

### 8.1.3 与其他预算样本的边界区分

#### C. 已进入申请冻结期
- 应表现为冻结期语义
- 不应冒充组织异常语义

#### D. 已到期
- 应表现为已到期语义
- 不应冒充组织异常语义

## 8.2 详情页：订单创建历史备注承接细化

### 8.2.1 目标
解决“同一条订单创建历史记录中，删除理由与失败原因/驳回说明并列冲突展示”的问题。

### 8.2.2 基本原则
订单创建历史中的每一条尝试记录，页面主展示语义必须以**该次尝试最终业务处置结果**为准，而不是简单并列堆叠所有过程信息。

### 8.2.3 承接优先级规则
对某一条历史记录，备注主语义按以下优先级处理：

#### 第一优先级：删除理由
当该次关联草稿已被业务主动删除时：

- 该条历史记录主展示应以**删除理由**为准；
- 不再同时主展示此前自动提交失败原因；
- 不再同时主展示此前审核驳回说明；
- 若存在失败原因/驳回说明，可作为过程背景保留在数据层，但页面主承接不得与删除理由并列冲突展示。

> 硬规则：  
> **只要该次草稿尝试最终状态为“已被业务主动删除”，则该条历史记录展示语义以删除理由为准。**

#### 第二优先级：审核驳回回草稿说明
当该次尝试不是被删除，而是“已创建后在客服/财务审核中被驳回回草稿”时：

- 该条历史记录主展示可承接“审核驳回回草稿”的说明语义；
- 外层摘要按“草稿”承接；
- 备注表现为说明性文本，不额外叠加无关警告标题。

#### 第三优先级：自动提交失败原因
当该次尝试是自动提交失败，且草稿仍保留、业务后续选择在原草稿基础上继续编辑时：

- 该条历史记录主展示保留失败原因；
- 后续若再产生新尝试，则通过历史序列承接；
- 不得提前用“删除理由”覆盖，因为该草稿并未删除。

### 8.2.4 三类固定业务场景承接

#### 场景①：原草稿继续编辑并再次尝试发起
要求：

1. 失败记录保留失败原因或驳回回草稿说明；
2. 后续重试生成新记录；
3. 若最终成功：
   - 外层只显示最新成功结果；
   - 不继续显示失败主提示；
   - 历史展开区保留之前失败记录；
4. 截图 3 类样式逻辑继续承接，但需与本次优先级规则一致。

#### 场景②：删除原草稿，不再发起
要求：

1. 删除原草稿必须填写工单关联草稿删除理由；
2. 该条对应历史记录主展示只显示删除理由；
3. 不得再同时堆叠显示原自动提交失败原因；
4. 外层若该组最终业务放弃：
   - 只展示最新删除理由或等价业务放弃备注；
   - 不展示“当前为草稿状态”等警告标题；
5. 截图 2 类现有“一条失败备注 + 一条删除备注并存”的效果必须修正。

#### 场景③：删除原草稿，并新建一份草稿重新关联工单号继续发起
要求：

1. 被删除的旧草稿记录：
   - 只显示删除理由；
   - 不再主展示旧失败原因；
2. 新建的草稿记录：
   - 按实际结果展示；
   - 若创建成功，则显示“已创建”；
   - 若再次失败，则按失败原因或驳回说明承接；
3. 若新增草稿最终成功：
   - 外层按成功结果展示；
   - 历史展开区可保留旧删除记录 + 新成功记录；
   - 不得因为旧草稿曾失败，就在外层继续显示失败主提示。

### 8.2.5 外层卡片承接要求
外层卡片仍保持“只承接最新结果摘要”的总原则，但需补充以下细则：

1. **最终已创建**：
   - 外层显示“已创建”
   - 不显示失败主提示
   - 旧删除理由、旧失败原因仅保留在展开区

2. **最终业务放弃**：
   - 外层只显示最新删除理由或最新业务放弃备注
   - 不得堆叠两条以上主备注
   - 不显示“订单当前为草稿状态”等警告标题

3. **审核驳回回草稿且未删除**：
   - 外层状态按“草稿”承接
   - 只展示回草稿说明性备注
   - 不堆叠其他无关失败提示

### 8.2.6 展开区承接要求
订单创建历史展开区可展示每次尝试的：

- 时间
- 订单编号 / 草稿编号
- 状态
- 对应备注或失败原因

但须遵循：

1. 单条记录只能有一个主备注语义；
2. 若该条记录最终被删除，则主备注为删除理由；
3. 若该条记录未删除但失败，则主备注为失败原因；
4. 若该条记录是审核驳回回草稿，则主备注为驳回回草稿说明；
5. 不得把删除理由与失败原因并列成两条同层主备注。

## 8.3 详情页 Mock 场景补强

### 8.3.1 目标
让详情页 mock 不仅有“主场景”，还能够明确表达“子场景差异”。

### 8.3.2 本轮最少补强内容
至少需补强以下子场景语义：

1. 自动提交失败 → 原草稿继续编辑重试
2. 自动提交失败 → 删除原草稿并业务放弃
3. 自动提交失败 → 删除原草稿后新建草稿并最终成功
4. 已创建后审核驳回回草稿 → 原草稿继续编辑重试
5. 已创建后审核驳回回草稿 → 删除原草稿并业务放弃
6. 已创建后审核驳回回草稿 → 删除原草稿后新建草稿并再次成功

### 8.3.3 实现方式要求
优先采用以下方式补强：

- 在现有 scene key 基础上，补充 group 级别的场景语义注释；
- 在规则文档中明确每个 group 的验证目的；
- 在 mock 数据中通过 remark / failReason / deleteReason / history 注释等方式体现差异；
- 只有当现有 scene 已无法承接时，才新增独立 scene key。

### 8.3.4 推荐承接对象
优先补强现有如下场景：

- `completed-retry`
- `completed-all-failed`

必要时可补充新的独立 scene，但必须在文档中同步：

- scene key
- id
- 工单号
- 主状态
- 核心验证点
- 不应出现的错误
- 推荐访问方式

### 8.3.5 Mock 注释要求
`mocks/index.ts` 中与详情页相关的 mock，必须补充：

- 每个关键 group 的业务语义说明；
- 哪一组是“继续编辑重试”；
- 哪一组是“删除后放弃”；
- 哪一组是“删除后新建草稿并成功”；
- 哪一组是“审核驳回回草稿”；
- 当前记录的备注应以删除理由、失败原因还是驳回说明为主。

## 11.1 预算选择页验收标准

### 通过项
1. 页面存在“`isAbnormal=true` 但仍可选”的正式样本；
2. 该样本仍可点击选择；
3. 页面仅提示异常，不阻断使用；
4. 显示字段口径仍为：
   - `剩余额度（净额）`
   - `预算总额（净额）`
5. 不将该样本误标为冻结期、已到期、额度不足或不可申请。

### 不通过项
1. `isAbnormal=true` 被实现为禁选；
2. 页面无此正式样本；
3. 用旧口径“可用金额”“总额”替代净额口径；
4. 将异常预算与 `applicationAvailable=false` 混为一类。

---

## 11.2 详情页订单创建历史验收标准

### 通过项
1. 原草稿继续编辑重试时，历史记录仍可展示原失败原因；
2. 删除原草稿且不再发起时，该条记录只显示删除理由；
3. 删除原草稿并新建草稿成功时：
   - 旧记录显示删除理由；
   - 新记录显示成功结果；
   - 外层只显示成功摘要；
4. 最终业务放弃时，外层只显示最新删除理由或业务放弃备注；
5. 展开区单条记录不再同时堆叠“失败原因 + 删除理由”两条主备注；
6. 最终已创建时，外层不再保留失败主提示。

### 不通过项
1. 被删除草稿仍以失败原因作为主备注；
2. 同一条历史记录同时主展示失败原因和删除理由；
3. 最终成功外层仍显示失败提示；
4. 业务放弃场景外层堆叠多条备注；
5. 旧截图 2 的问题样式仍存在。
