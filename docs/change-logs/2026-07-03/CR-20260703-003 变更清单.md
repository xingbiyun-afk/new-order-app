# CR-20260703-003 产品申请工单详情页驳回态与重新发起链路优化

## 1. 变更背景

截至 `2026-07-03`，以下内容已完成并收口：

- `CR-20260703-001 产品申请工单选择链路与驳回后重新发起体验优化`
- `CR-20260703-002 产品申请工单创建链路收尾与样式口径统一补充`

当前产品申请工单的创建链路已进入相对稳定状态，已完成：

1. 预算选择页、创建页、产品选择页、专卖店选择页与批量导入能力的当前阶段收口；
2. 冻结期驳回后重新发起在**创建页承接侧**的原预算号沿用、当前场景下不允许重新选择预算号、提示与校验逻辑；
3. 非冻结期驳回后重新发起按普通新建发起（方案 B）的当前阶段承接；
4. 创建页、预算页等页面中金额与预算文案口径的统一；
5. 专卖店选择页继续沿用 App 原页面能力承接的补漏收口。

在此基础上，详情页成为下一阶段最关键的未收口页面，尤其在以下方面仍需继续优化：

1. 驳回态信息展示层级不够明确，用户不易快速理解“为什么被驳回、接下来能做什么”；
2. “重新发起”入口在详情页中的展示条件、交互限制、前置判断与提示表达尚未正式收口；
3. 详情页与创建页之间关于驳回后重新发起的链路，仍需形成用户视角的一致体验；
4. 详情页中的金额文案、状态提示、附件区表达等仍需与已收口页面保持一致；
5. 当前详情页 mock 数据不足以充分展示产品申请工单的完整链路、状态差异、多个产品申请订单明细、审批流、附件、订单结果与异常场景；
6. 当前项目仍存在 `uuid moderate` 依赖安全遗留问题，需要继续推进处理路径评估，但不适合以高风险方式在本轮强制修复。

因此，需新增本轮变更，以“详情页驳回态与重新发起链路”作为主线进行收口，并一并纳入少量强相关页面补丁项、mock 数据补强项与技术遗留推进项。

---

## 2. 变更目标

本轮目标如下：

1. 优化详情页在“已驳回”场景下的信息展示层级，使用户能快速理解驳回原因与当前状态；
2. 收口“重新发起”入口在详情页中的展示、点击、限制与提示逻辑；
3. 统一详情页与创建页在驳回后重新发起场景中的承接口径；
4. 补齐详情页中的金额展示、附件区、状态提示等强相关体验细节；
5. 补强详情页 mock 数据，确保能覆盖产品申请工单的主要状态、主要区块与主要异常场景；
6. 对详情页字段与区块边界做一轮轻量梳理，为后续正式详情页规则文档建档打基础；
7. 持续推进 `uuid moderate` 依赖安全遗留问题，形成下一步路径评估与书面沉淀，但不以本轮彻底修复为目标。

---

## 3. 变更范围

本轮变更范围包括：

1. 产品申请工单详情页；
2. 详情页中的驳回态展示与“重新发起”入口；
3. 详情页与创建页在重提场景下的承接关系；
4. 详情页中的金额展示、附件区与状态提示；
5. 详情页 mock 数据与详情页相关演示数据结构；
6. 详情页字段与区块边界的轻量梳理；
7. `exceljs / uuid` 依赖安全遗留的持续评估与记录；
8. 与上述内容相关的正式文档回写。

## 4. 详情页驳回态信息展示优化

### 4.1 本轮目标
提升“已驳回”状态下详情页的信息可理解性，使用户在进入页面后，能快速识别：

1. 当前工单已被驳回；
2. 驳回原因是什么；
3. 是否允许重新发起；
4. 若允许重新发起，后续动作是什么；
5. 若不允许重新发起，不允许的原因是什么。

### 4.2 本轮重点优化方向
包括但不限于：

1. 驳回状态标签在详情页中的展示层级优化；
2. 驳回原因展示位置前置或视觉突出；
3. 驳回时间、审批节点、审批备注等与驳回原因的层级关系优化；
4. 用户进入详情页首屏或关键区块时，不应需要过多滚动才能理解“已驳回 + 原因 + 后续动作”。

### 4.3 本轮边界
1. 本轮不展开全部状态下的详情页统一改版；
2. 本轮优先围绕“已驳回”场景收口；
3. 其他状态下若顺带发生轻量样式统一，可纳入，但不扩展为全状态重构。

## 5. 重新发起入口与前置判断优化

### 5.1 本轮目标
在详情页中收口“重新发起”入口逻辑，使其在用户视角上形成明确、可预期、可解释的交互。

### 5.2 本轮需明确的内容
包括但不限于：

1. 哪些情况下详情页显示“重新发起”入口；
2. 哪些情况下不显示；
3. 哪些情况下显示但不可点击；
4. 点击“重新发起”前需做哪些前置判断；
5. 到期、冻结期、非冻结期等场景下的入口提示如何表达；
6. 页面需避免出现“能看到入口，但不知道能不能用”的模糊状态。

### 5.3 与当前已确认规则的关系
需严格承接已有确认口径：

1. 冻结期原单重提：
   - 创建页沿用原预算号；
   - 当前场景下**不允许重新选择预算号**；
   - 提交时仍按当前剩余额度校验；
   - 到期后不可继续基于原单重提。
2. 非冻结期驳回后重新发起：
   - 当前阶段按方案 B 处理即可；
   - 即按普通新建发起，用户自行重选预算。

### 5.4 重要口径约束
本轮不得引入“预算锁定”作为新的预算状态或新的业务概念。  
相关场景统一表达为：

- 原预算号沿用；
- 当前场景下不允许重新选择预算号；
- 预算号不可切换；
- 不允许重新进入预算选择页。

不得将其表达为：

- 预算锁定；
- 锁定状态；
- 预算被锁定；
- 新增预算状态。

### 5.5 本轮边界
1. 本轮可以完善详情页入口展示与前置判断；
2. 但不展开后端流程重构；
3. 真实接口与状态流转细节，可继续用 mock / 占位方式承接，但前端表现与规则口径必须一致。

## 6. 详情页与创建页重提承接口径统一

### 6.1 本轮目标
确保用户从详情页点击“重新发起”进入创建页时，前后页面在以下内容上保持一致：

1. 为什么能重新发起；
2. 为什么不能重新发起；
3. 原预算号是否沿用；
4. 当前场景下是否允许切换预算号；
5. 进入创建页后看到的提示与详情页中的提示不冲突。

### 6.2 需统一的重点方向
包括但不限于：

1. 冻结期场景下的说明提示；
2. 已到期场景下的不可继续重提提示；
3. 非冻结期按普通新建处理时的入口说明；
4. 详情页 CTA 文案与创建页内提示文案的统一；
5. 如用户从详情页重提进入创建页，页面提示应能解释当前状态，而不是只依赖创建页逻辑自行兜底。

## 7. 详情页金额与文案口径统一补丁

### 7.1 纳入原因
当前创建页、预算页、产品页的金额与预算文案口径已基本统一。  
为避免详情页体验割裂，本轮可同步补齐详情页中的同类口径。

### 7.2 本轮处理方向
包括但不限于：

1. 金额展示统一采用：
   - 千分符
   - 两位小数
2. 涉及预算相关文案时，统一采用：
   - `剩余额度（净额）`
   - `预算总额（净额）`
3. 分组金额、整单金额、结果金额、预算提示等同类表达与创建页保持一致；
4. 需检查是否仍存在：
   - `可用金额`
   - `总额`
   - 未统一金额格式
   等旧口径残留。

### 7.3 本轮边界
本轮只做强相关字段与展示口径统一，不单独将详情页扩展为一轮“全量金额体系重构”。

---

## 8. 详情页附件区与状态提示优化

### 8.1 纳入原因
详情页中“附件查看”“驳回提示”“异常提示”“结果失败提示”均属于用户最关心的内容之一，与驳回态与重新发起链路强相关，适合本轮一并收口。

### 8.2 本轮处理方向
包括但不限于：

1. 附件区的预览 / 下载入口更明确；
2. 驳回原因提示、失败原因提示、异常提示的层级更清晰；
3. 到期、当前场景下不允许重新发起等提示需做到“可理解”，避免仅展示技术性判断结果；
4. 状态提示表达需尽量统一，不引入新的口径分裂。

### 8.3 本轮边界
1. 不重做附件系统；
2. 不新增复杂的附件操作；
3. 仅围绕详情页阅读体验、状态理解与后续动作引导优化。

---

## 9. 详情页 mock 数据补强与场景覆盖

### 9.1 纳入原因
详情页后续将承接驳回态展示、重新发起入口、审批流、附件、订单结果与异常提示等多类信息。  
若仅使用单一简单 mock 数据，无法充分验证页面结构、状态分支与用户体验，因此本轮需同步补强详情页 mock 数据。

### 9.2 本轮目标
通过 mock 数据覆盖产品申请工单详情页的主要状态、主要区块与主要异常场景，确保页面优化具备可验证基础。

### 9.3 建议覆盖的场景
包括但不限于：

1. **处理中场景**
   - 有基础申请信息；
   - 有审批流节点；
   - 有多个产品申请订单明细。
2. **已驳回且允许重新发起场景**
   - 冻结期原单重提场景；
   - 原预算号沿用；
   - 当前场景下不允许重新选择预算号；
   - 页面提示明确。
3. **已驳回但不允许继续基于原单重提场景**
   - 例如原预算已到期；
   - 页面需明确提示不可继续重提。
4. **已驳回且按普通新建重新发起场景**
   - 非冻结期；
   - 可重新发起；
   - 但不强制承接原预算号。
5. **已结束场景**
   - 审批通过；
   - 正式订单结果可展示；
   - 能体现不同客户层级对应的不同订单类型。
6. **多产品申请订单明细场景**
   - 一张工单包含多个产品申请订单明细；
   - 能展示不同订单归属专卖店；
   - 能展示多个产品明细。
7. **多结果订单场景**
   - 部分分组成功；
   - 部分分组失败；
   - 有失败原因展示。
8. **多附件场景**
   - 多个附件；
   - 不同文件类型；
   - 支持预览 / 下载承接展示。
9. **审批流较完整场景**
   - 发起节点；
   - 审批通过节点；
   - 驳回节点；
   - 备注与时间。

### 9.4 边界说明
本轮 mock 数据补强用于支撑详情页参考实现与验收，不代表正式接口协议与生产数据结构已最终定稿；真实环境仍以后续接口与正式联调结果为准。

## 11. 已知技术遗留推进：uuid moderate 依赖安全问题

### 11.1 纳入原因
当前 `uuid` moderate 告警已在 `CR-20260703-002` 中完成首轮风险识别、影响判断与“不采用 \`npm audit fix --force\`”的说明。  
下一步不应仅停留在“记录遗留”，还需逐步推进后续工程化处理路径。

### 11.2 本轮目标
本轮不要求彻底消除 `uuid` moderate 告警，但需继续推进以下事项：

1. 复核当前 `npm audit` 结果是否有变化；
2. 复核 `exceljs` 上游是否已有依赖修复进展；
3. 评估 `exceljs` 在当前参考实现与后续正式工程中的保留必要性；
4. 明确后续可选路径，例如：
   - 路径 A：继续保留 `exceljs`，等待上游修复；
   - 路径 B：正式工程阶段替换为其他 Excel / CSV 方案；
   - 路径 C：弱化 Excel 文件导入依赖，仅保留文本 / CSV 能力；
5. 在开发总结中补充“下一步处理路径建议”，使该遗留问题从“已识别”进入“已形成推进方案”的阶段。

### 11.3 本轮边界
1. 不以“清零 moderate 告警”为本轮硬性交付目标；
2. 不因清理告警而执行高风险 `npm audit fix --force`；
3. 不要求本轮立即替换现有 Excel 解析库；
4. 本轮重点是**继续评估、形成路径、沉淀建议**。
