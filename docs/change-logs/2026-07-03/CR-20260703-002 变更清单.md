# CR-20260703-002 产品申请工单创建链路收尾与样式口径统一补充

## 1. 变更背景

在以下内容已基本收口后：

- `CR-20260702-001 产品申请工单发起页预算选择页面样式与体验优化`
- `CR-20260702-002 产品申请工单发起页面样式与体验优化`
- `CR-20260703-001 产品申请工单选择链路与驳回后重新发起体验优化`

当前“产品申请工单”创建链路已完成预算选择页、创建页、产品选择页、专卖店选择页、批量导入能力与驳回后重新发起承接的主体参考实现，但在验收与回看过程中，仍存在以下需继续收口的问题：

1. 创建页相关文案、提示、金额与预算联动口径仍需再统一；
2. 批量导入模板已调整为无表头格式，但对应的导入说明、错误反馈与承接文案仍需补强；
3. 预算选择页与创建页在冻结期、到期、预算切换、不可选原因等场景下的提示口径仍需进一步统一；
4. 专卖店选择页虽已做视觉回归，但尚未基于 App 现有页面 / AI 参考截图做到尽量 1:1 承接，仍存在实现偏差；
5. 当前依赖安全仍遗留 `uuid` moderate 漏洞，需要正式完成来源、影响与后续处理路径的评估记录；
6. 当前阶段希望优先将创建链路相关遗留问题集中收尾，为下周进入详情页优化与改造节奏腾出空间。

因此，需新增本轮变更，对创建链路相关页面、文案、联动提示、导入能力与技术遗留进行统一收尾与补充。

---

## 2. 变更目标

本轮目标如下：

1. 将创建页当前已知遗留问题与体验细节继续收口；
2. 补强批量导入模板调整后的页面说明、导入反馈与规则承接；
3. 统一预算选择页与创建页之间的预算展示、状态提示、不可选原因与切换提示口径；
4. 将专卖店选择页按 App 现有页面与 AI 参考截图进一步回归，尽量做到 1:1 承接；
5. 对当前 `uuid` moderate 依赖安全遗留完成正式评估与风险记录；
6. 通过本轮补充，使创建链路进入相对稳定状态，便于后续将重点转入详情页优化与改造。

---

## 3. 变更范围

本轮变更范围包括：

1. 产品申请工单创建页；
2. 产品申请工单预算选择页与创建页的预算承接联动；
3. 产品申请工单专卖店选择页；
4. 产品申请工单批量导入弹窗、模板下载与导入反馈承接；
5. 与创建链路相关的金额展示、提示文案与页面说明；
6. 当前 Excel 导入相关依赖的安全遗留评估记录；
7. 与上述内容相关的正式文档回写。

## 4. 创建页体验与口径收尾

### 4.1 创建页本轮定位
本轮继续以“创建链路收尾”为目标，不重新设计创建页主体结构，不推翻既有规则文档，而是在已实现基础上继续补齐细节、统一口径并收口遗留问题。

### 4.2 需继续收口的方向
包括但不限于：

1. 页面内提示文案是否与当前正式业务口径一致；
2. 金额、剩余额度（净额）、分组小计、整单金额等展示是否统一采用：
   - 千分符
   - 两位小数
3. 创建页中与预算相关的提示文案是否统一采用“剩余额度（净额）”口径；
4. 已选产品返显、数量提示、预算关系提示、附件必传提示等是否仍存在细节不一致问题；
5. 当前页可直接判断的缺失项前置禁用与提示是否足够清晰；
6. 长文本返显、标题截断、完整查看能力是否满足移动端使用体验。

### 4.3 本轮处理原则
1. 以现有创建页正式规则文档为基础，不引入新的业务对象或旧概念；
2. 优先修正体验不一致、文案不统一、提示不清晰的问题；
3. 不将本轮扩展为新的创建页结构重做；
4. 不因局部视觉调整影响已确认的提交、预算承接与分组结构规则。

## 5. 批量导入能力补强与规则收口

### 5.1 本轮背景
`CR-20260703-001` 已完成批量导入模板由“含表头”调整为“无表头”格式，并完成导入解析能力与 `exceljs` 改造。  
但模板格式变化后，页面说明、帮助文案、导入错误反馈与用户理解成本仍需继续收口。

### 5.2 本轮补强目标
本轮需补强以下内容：

1. 明确“模板第 1 行即为数据，无表头”的页面说明；
2. 明确固定字段顺序：
   1. 产品编码（产品编号）
   2. 数量
   3. 专卖店编号
3. 文本导入与文件导入在字段顺序说明、格式要求、错误反馈上保持一致；
4. 导入弹窗中的说明文案、示例文案、错误提示与模板下载文案保持一致；
5. 导入后承接到创建页时的结果提示与页面反馈更清晰；
6. 在不改变覆盖导入既有规则的前提下，强化用户对“覆盖导入”结果的理解。

### 5.3 本轮明确不重做的内容
本轮不重做以下内容：

1. 不重做覆盖导入业务规则；
2. 不新增新的导入格式；
3. 不改变当前文件类型范围；
4. 不重做文本导入与文件导入的底层校验体系；
5. 不将批量导入单独拆出新的正式规则明细文档，仍并入创建页规则文档承接。

---

## 6. 预算选择页与创建页联动补强

### 6.1 本轮目标
本轮需继续收口预算选择页与创建页之间的联动体验，确保用户在预算选择、回填、切换与提交前的预算感知口径一致。

### 6.2 需统一的重点方向
包括但不限于：

1. “剩余额度（净额）”文案在预算选择页、创建页摘要区、滚动态预算摘要条、预算关系提示中的统一；
2. 预算到期、申请冻结中、不可选、零剩余额度等状态在创建页与预算选择页中的提示口径统一；
3. 预算切换时的确认文案、清空范围提示与结果反馈统一；
4. 提交前预算校验失败时的提示口径与前置提示保持一致；
5. 驳回后重新发起场景下，创建页对预算承接、锁定与不可切换提示的表达继续收口；
6. 不可选原因需尽量让用户可理解，避免只展示技术性字段或模糊提示。

### 6.3 本轮边界
1. 不重新定义预算选择页已确认的业务规则；
2. 不改变预算标签优先级、排序规则、冻结期特例规则；
3. 不改变“草稿 / 已作废 / 已到期不展示”“零剩余额度展示但不可选”“`applicationAvailable=false` 不可选”等既定规则；
4. 本轮重点是联动体验与口径统一，不是重做预算规则体系。

---

## 7. 专卖店选择页按 App 现有页面 1:1 回归补漏

### 7.1 问题说明
此前已明确：

- 专卖店选择页直接沿用 App 现有页面与功能；
- 不单独新增专卖店选择页面规则明细文档；
- 业务过滤口径与回填逻辑保持不变；
- 本应按 App 现有页面 / AI 参考截图进行尽量 1:1 承接。

但当前实现虽已完成一轮视觉统一，仍未基于参考截图与现有 App 页面做到足够接近，存在“风格接近但并非 1:1 回归”的偏差，因此需补漏收口。

### 7.2 本轮要求
本轮需按 App 现有页面与 AI 参考截图，尽量完成以下内容的 1:1 回归：

1. 搜索区结构、样式、间距、占位文案、清空交互；
2. 列表项排版、信息层级、点击反馈、选中反馈；
3. 空态与搜索无结果态的结构、文案、视觉表达；
4. 顶部导航区、返回区、列表容器与卡片间距；
5. 页面整体视觉密度与现有 App 页面保持一致。

### 7.3 明确保持不变的内容
以下内容本轮保持不变：

1. 搜索范围；
2. 客户层级 0-6 可搜、排除层级 5 的业务口径；
3. 权限范围控制；
4. 启用下单接口返回允许下单的过滤逻辑；
5. 回填逻辑；
6. 创建页中同店编多产品明细的分组承接规则。

### 7.4 文档边界
本轮仍不单独新增“专卖店选择页面规则明细文档”。  
仅在创建页规则文档及相关引用文档中补充其入口、筛选口径、承接边界与“沿用 App 现有页面”的说明。

## 8. 依赖安全遗留评估与风险记录

### 8.1 本轮纳入原因
当前创建链路中的批量导入能力已完成 `xlsx -> exceljs` 替换与按需加载，但依赖安全仍遗留 `uuid` moderate 漏洞。该问题虽不适合作为本轮强制修复项，但需要完成正式评估与记录，避免后续失焦。

### 8.2 本轮要求
本轮需对该遗留项完成以下评估与沉淀：

1. 复核当前 `npm audit` 结果；
2. 明确漏洞来源包、依赖层级与影响范围；
3. 明确该漏洞是否来自当前业务直接依赖；
4. 明确 `npm audit fix --force` 对 `exceljs` 版本的影响；
5. 明确为何本轮不采用强制自动修复方案；
6. 在开发总结中沉淀当前判断、风险结论与后续建议。

### 8.3 当前处理原则
1. 本轮可纳入“评估、复核、记录、说明”；
2. 本轮不以“将 moderate 告警全部清零”为硬性交付目标；
3. 不为消除当前 `uuid` moderate 告警而强制降级 `exceljs`；
4. 若后续要彻底处理，应在更合适的时机专项评估依赖升级或 Excel 解析库替换方案。

## 10. 开发与验收要点

### 10.1 创建页
1. 金额展示统一为千分符 + 两位小数；
2. 预算相关文案统一为“剩余额度（净额）”；
3. 页面内提示、返显、禁用态与提交校验提示口径一致；
4. 不引入已废弃旧概念；
5. 不破坏当前分组结构、预算承接与提交流程。

### 10.2 批量导入
1. 模板说明、弹窗说明、错误提示、下载模板口径一致；
2. 明确“无表头，第 1 行即为数据”；
3. 覆盖导入规则不变，但需让用户能理解；
4. 文件导入与文本导入在页面说明层面保持统一。

### 10.3 预算联动
1. 创建页与预算选择页的预算口径统一；
2. 到期、冻结、不可选、零剩余额度等提示清晰；
3. 切换预算的确认提示与清空影响范围表达一致；
4. 驳回后重提预算锁定提示与页面行为保持一致。

### 10.4 专卖店选择页
1. 必须以 App 现有页面 / AI 参考截图为基准，不仅是“视觉风格接近”；
2. 应尽量做到结构、间距、反馈与空态表达的 1:1 承接；
3. 不修改既有业务过滤与回填规则。

### 10.5 依赖安全遗留评估
1. 必须形成书面结论；
2. 必须写入开发总结；
3. 必须明确本轮不采用 `npm audit fix --force` 的原因；
4. 必须保留后续处理建议。
