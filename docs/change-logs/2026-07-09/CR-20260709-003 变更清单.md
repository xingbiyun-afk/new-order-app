## 2. 变更背景

当前项目经过前序 CR 收口后，创建页与详情页主结构、主规则、主展示模型已基本稳定。  
但以下场景仍缺少明确制度化说明，容易导致后续 AI、前端、联调、验收出现理解偏差：

1. 工单提交后，审批任一环节长期无人处理时，缺少明确的**系统自动驳回规则**
2. 审批过程中，若相关组织信息发生变化，原属于该组织范围的在途工单缺少明确的**处理闭环**
3. “财务通知相关审批人驳回”与“超过 14 天系统自动驳回”之间的关系尚未文档化
4. 详情页中对这类驳回的**状态承接、驳回原因文案、审批流节点展示**尚未固定
5. 这些规则与审批流程管理、审批异常监控、总状态规则之间缺少联动说明

因此，本轮需要以“小切口、规则补增说明型 CR”的方式，对上述场景进行文档治理与规则回写。

## 3. 本轮变更目标

本轮目标不是新增复杂系统能力，而是把以下内容明确下来：

1. **审批超时自动驳回规则**
   - 明确起算点、适用范围、触发条件、处理结果、页面承接口径

2. **组织变更场景在途工单驳回规则**
   - 明确组织变更后谁通知、谁处理、多久无人处理时系统如何兜底

3. **财务通知驳回与系统自动驳回的关系**
   - 明确人工处理优先还是系统自动处理优先
   - 明确两者在页面展示上的差异

4. **详情页承接规则**
   - 明确状态展示
   - 明确驳回原因口径
   - 明确审批流中如何表达“人工驳回 / 超时自动驳回 / 组织变更后驳回”

5. **相关规则文档联动回写**
   - 让后续 AI、设计、开发、测试、验收都能以统一文档为准执行

## 5. 本轮纳入范围

本轮**纳入**以下内容：

### 5.1 审批超时自动驳回规则补增
需明确：
1. 适用对象：产品申请工单审批中的任一审批环节
2. 超时阈值：**14 天**
3. 触发条件：某审批环节自进入待审批起，连续超过 14 天无人审批操作
4. 处理结果：系统自动驳回工单
5. 页面承接：详情页需展示为驳回结果，并说明为超时自动驳回

### 5.2 组织变更场景在途工单处理规则补增
需明确：
1. 审批过程中若相关组织信息发生变更
2. 变更前属于该组织范围的在途工单，不再按原链路继续长时间悬置
3. 由**财务通知相关环节审批人驳回**
4. 若通知后仍超过 14 天无人操作，系统仍按超时规则自动驳回
5. 详情页需能区分：
   - 人工驳回（组织变更后按通知驳回）
   - 系统超时自动驳回（组织变更后仍无人处理）

### 5.3 财务通知驳回与系统自动驳回关系说明
需明确：
1. 财务通知本身不是系统审批动作
2. 财务通知的目的是推动相关审批人执行驳回
3. 若审批人在 14 天内完成驳回，则按**人工驳回**承接
4. 若 14 天内仍无人审批，则按**系统自动驳回**承接
5. 两者不可混淆展示为同一种原因

### 5.4 详情页状态 / 驳回原因 / 审批流展示承接
需明确：
1. 详情页主状态仍按现行规则承接，不新增复杂状态体系
2. 这两类场景最终在详情页主状态层统一承接为：
   - **已驳回**
3. 但驳回原因与审批流说明需区分来源：
   - 普通审批驳回
   - 审批超时自动驳回
   - 组织变更后人工驳回
   - 组织变更后超时自动驳回
4. 审批流节点文案、详情页驳回信息区、历史记录区要保持一致口径

## 7. 核心规则口径（本轮固定）

以下为本轮必须落库的核心规则，AI 执行时不得自行发挥成别的含义。

### 7.1 审批超时自动驳回规则

1. 工单提交进入审批流程后，**任一审批环节**若连续超过 **14 天**无审批操作，系统自动驳回该工单
2. “14 天”的判定口径应按**进入当前待审批节点时刻**起算
3. 若节点期间已发生审批动作（同意 / 驳回 / 其他有效审批处理），则不触发该次自动驳回
4. 自动驳回后的工单主状态承接为：**已驳回**
5. 自动驳回属于**系统处理结果**，不是人工审批意见
6. 详情页驳回原因中需明确体现“审批超时自动驳回”语义
7. 审批流中需能看出该结果来源于系统超时处理，而非审批人主动驳回

### 7.2 组织变更场景在途工单处理规则

1. 若工单已提交且仍处于审批中
2. 若审批过程中相关组织信息发生变更
3. 对于变更前属于该组织范围的在途工单，由**财务通知相关环节审批人驳回**
4. 财务通知本身不改变工单状态
5. 只有当相关审批人执行驳回后，工单才按**人工驳回**落定
6. 若通知后相关环节在 **14 天**内仍无人审批，则系统自动驳回
7. 组织变更场景下也不新增新的主状态，最终仍统一归入：**已驳回**
8. 页面需体现“组织变更”是驳回背景原因之一，但不要扩展成复杂状态树

### 7.3 财务通知驳回与系统自动驳回的关系

1. 财务通知是**流程推动动作**，不是审批结果
2. 审批人收到通知后执行驳回，属于**人工驳回**
3. 若审批人未处理且超时，则属于**系统自动驳回**
4. 不得把“已通知财务”或“财务已通知审批人”展示成审批结果
5. 页面展示时，需区分：
   - 驳回执行者是审批人
   - 驳回执行者是系统
6. 若有审批流文案，应明确区分：
   - “审批人驳回”
   - “系统超时自动驳回”

### 7.4 详情页状态承接规则

1. 不新增新的主状态枚举
2. 涉及本轮两类场景时，详情页主状态统一承接为：**已驳回**
3. 驳回信息区需补充更明确的原因类型说明
4. 审批流区需补充节点来源说明
5. 订单结果区不因为本轮新增规则而重做结构
6. 预占库存订单展示边界不因本轮规则而改变
7. 已收口的详情页阅读路径优化结论不回退

### 7.5 驳回原因口径建议

本轮建议固定以下分类语义，用于文档承接与后续页面表达：

1. **普通审批驳回**
   - 由审批人基于业务判断主动驳回

2. **审批超时自动驳回**
   - 由系统因审批节点超时无操作自动驳回

3. **组织变更后人工驳回**
   - 因组织变更，财务通知后，由相关审批人执行驳回

4. **组织变更后超时自动驳回**
   - 因组织变更后仍无人处理，超过 14 天后由系统自动驳回

说明：
- 上述属于**驳回原因 / 来源分类**
- 不等同于新增主状态
- 不等同于新增预算状态
- 不等同于新增订单结果分类
