## 2. 变更定位

本 CR 属于**规则补增说明型小切口**，用于在当前创建页与详情页主结构、主交互、主展示模型已基本收口的前提下，进一步把“后续最容易误解、最容易回退、最容易在正式工程承接中走偏”的规则边界补写清楚。

本 CR 的核心定位是：

1. **补清参考实现与正式工程承接边界**
2. **补清历史快照字段与实时主数据字段口径**
3. **补清预算并发校验与提交失败回退说明**
4. **补清导入重复 / 冲突 / 覆盖处理说明**
5. **补清后续 AI 与开发协作中的承接约束**

本 CR 必须明确为：

1. **不是新增业务功能**
2. **不是新增业务规则主线**
3. **不是审批超时自动驳回专项**
4. **不是组织变更场景在途工单审批专项**
5. **不是接口协议专项设计**
6. **不是正式后端联调方案设计**
7. **不是审批流系统级异常设计**
8. **不是附件生命周期专项**
9. **不是列表页承接专项**
10. **不是创建页或详情页结构改版**
11. **不是 mock 重构专项**
12. **不是把已有规则拆成更多独立文档**

本轮目标是在**不增加大量新文档、不制造新的阅读入口、不扩大开发范围**的前提下，把当前已经成立但尚未在现有规则文档中系统说透的“承接层规则”补齐，使后续 AI、开发、测试、联调同事能够：

- 明确哪些规则已经是现行正式执行依据
- 明确哪些行为只是参考实现阶段的轻量承接方式
- 明确哪些正式工程仍需继续承接
- 明确哪些字段应按历史事实理解，哪些字段可按当前结果理解
- 明确预算额度与导入行为在边界场景下的解释方式
- 避免把当前已收口规则无意识改回旧口径

## 3. 变更背景

### 3.1 当前页面主规则已基本稳定，但“承接层说明”仍不够完整

截至目前，创建页与详情页的页面主规则已经通过多轮 CR 基本收口：

#### 创建页相关已完成内容
1. 创建页结构与命名口径收口
2. 预算选择页体验与冻结期承接优化
3. 专卖店 / 产品选择页承接优化
4. 批量导入能力与模板口径收口
5. 驳回后重新发起承接优化
6. 创建页场景验收样本、返回/回填/store 边界、错误示例、导入链路样本已在 `CR-20260709-001` 中补齐

#### 详情页相关已完成内容
1. 三类主状态收口：处理中 / 已驳回 / 已结束
2. 详情页结构顺序与区块层级收口
3. 审批流区与订单结果区展示模型收口
4. 订单结果区按专卖店结果单元展示
5. 当前结果说明 5 类口径固定
6. `draftId` 聚合草稿链路与历史阅读路径收口
7. 详情页规则文档与场景样本已基本补齐

也就是说，当前主要问题已不再是：

- 页面有没有做
- 功能能不能演示
- 结构是不是搭起来了

而更集中在：

- 哪些规则是现行正式规则
- 哪些行为是参考实现轻量承接
- 哪些能力正式工程还要继续补
- 哪些字段应理解为历史快照
- 哪些字段可理解为当前结果
- 哪些边界不能再靠代码隐含

### 3.2 当前最容易反复争议的，不是页面主功能，而是“承接层边界”

当前最容易被误读、误改、误回退的点主要包括：

#### 3.2.1 参考实现 vs 正式工程承接边界未系统写清
例如：

1. `sessionStorage` 是否会被误理解为正式草稿能力
2. route.query 回填是否会被误理解为正式稳定承接方案
3. 附件上传当前模拟实现与正式上传能力之间的关系
4. 文件导入本地解析与后续接口化承接之间的关系
5. 前端金额计算与后端最终校验的优先级关系

#### 3.2.2 历史快照字段 vs 实时主数据字段口径未统一收口
例如：

1. 详情页预算信息到底看提交时快照还是当前预算主数据
2. 产品名称 / JDE 价格 / 数量 / 金额的展示到底是历史事实还是实时值
3. 专卖店名称与实际订单中的专卖店关系如何理解
4. 审批流记录是历史审批事实还是当前组织结果
5. 订单结果中的实际生成信息与工单申请信息如何区分

#### 3.2.3 预算额度并发变化场景未单独说明
例如：

1. 页面看到的“剩余额度（净额）”与提交瞬间额度不一致
2. 填写过程中预算被其他申请占用
3. 提交瞬间预算不可用 / 已到期 / 已作废
4. 驳回重提沿用原预算号但额度不足
5. 提交失败后页面哪些信息保留、哪些允许继续调整

#### 3.2.4 导入重复 / 冲突 / 覆盖说明仍不够成体系
例如：

1. 同一店编下多行输入如何归并
2. 同店编同 SKU 多行数据的当前承接口径
3. 覆盖导入对现有页面状态的影响
4. 软警告导入与提交阻断的关系
5. 导入后再手工修改应以哪份数据为准

### 3.3 当前更适合继续做“小切口补增”，而不是开大专项

本轮不适合扩展为以下类型的 CR：

1. **接口 DTO / payload 设计专项**
2. **审批超时自动驳回专项**
3. **组织变更场景在途工单处理专项**
4. **审批异常监控联动专项**
5. **附件生命周期专项**
6. **列表页承接专项**
7. **正式工程联调方案专项**

原因很明确：

- 这些内容一旦展开，会迅速拉大范围；
- 会让当前 CR 从“现有规则文档补增说明”变成“跨系统方案设计”；
- 会增加文档入口数量；
- 会让 AI 和开发同事的执行注意力分散。

因此，本轮必须继续坚持：

- 只补现有核心文档
- 只补承接边界与解释口径
- 不额外制造新文档负担
- 不顺手扩大到审批链路新规则主线

## 4. 本轮核心目标

本轮核心目标固定为四类。

### 4.1 创建页补清“参考实现与正式工程承接边界”
明确以下内容：

1. `sessionStorage` 的定位仅为参考实现轻量承接，不等于正式草稿
2. route.query 的定位仅为当前创建链路返回与重提承接方式
3. 附件上传当前为模拟实现，正式工程需接入真实上传能力
4. 文件导入当前为前端本地解析，正式工程可接口化但不得破坏现有规则口径
5. 金额计算与预算前置校验属于前端辅助能力，正式提交以后端结果为准
6. 浏览器后退 query 丢失属于正式工程需继续承接的问题
7. 已收口规则不得在正式工程中被无变更单自行回退

### 4.2 创建页补清“预算并发校验与提交失败回退说明”
明确以下内容：

1. 页面展示的“剩余额度（净额）”仅为填写参考
2. 前端校验通过不等于后端提交必然成功
3. 提交瞬间若预算不足，应以后端结果为准
4. 提交瞬间若预算已到期 / 已作废 / 不可申请，应阻断提交
5. 提交失败后页面数据应保留，避免用户重新填写
6. 驳回重提沿用原预算号时仍应按当前预算可用性判断

### 4.3 创建页补清“导入重复 / 冲突 / 覆盖处理说明”
明确以下内容：

1. 同一店编多行数据如何承接
2. 同一店编 + 同一 SKU 多行输入如何理解
3. 不同店编同 SKU 如何承接
4. 覆盖导入对页面已有内容的影响
5. 导入后的软警告与提交阻断如何衔接
6. 导入后再手工修改时，以页面当前编辑结果为准

### 4.4 详情页补清“历史快照字段与实时主数据字段口径”
明确以下内容：

1. 预算信息的展示口径
2. 申请信息的展示口径
3. 产品信息的展示口径
4. 专卖店信息的展示口径
5. 审批流信息的展示口径
6. 订单结果信息的展示口径
7. 历史申请事实与当前系统结果之间的边界

## 8. 详细实施要求

### 8.1 对创建页规则文档的实施要求

#### 8.1.1 章节组织要求
新增内容必须优先以内嵌章节方式补入现有文档，不新拆独立专题文档。  
建议新增：

- `## 26. 参考实现与正式工程承接边界`
- `## 27. 预算并发校验与提交失败回退说明`

并在现有导入章节下新增：

- `24.x 导入重复、冲突与覆盖处理补充说明`

#### 8.1.2 “参考实现与正式工程承接边界”必须覆盖的点
至少覆盖以下 12 点：

1. 当前创建页为参考实现，不等同正式生产页面
2. `sessionStorage` 仅用于轻量承接
3. `sessionStorage` 不等于草稿
4. route.query 仅承接当前创建链路
5. 浏览器后退 query 丢失属于正式工程继续承接事项
6. 附件上传当前为模拟实现
7. 附件类型 / 大小 / 服务端返回仍需正式工程承接
8. 导入当前为本地解析
9. 正式工程可接口化但不得改变现有模板与校验口径
10. 金额计算是前端辅助能力
11. 提交以后端预算与金额校验结果为准
12. 已收口规则不得自行回退

#### 8.1.3 “预算并发校验与提交失败回退说明”必须覆盖的点
至少覆盖以下 12 点：

1. 页面额度仅供参考
2. 前端校验通过不代表后端必然成功
3. 填写中预算可能被占用
4. 提交瞬间可用金额可能不足
5. 提交瞬间预算可能到期
6. 提交瞬间预算可能作废
7. 提交瞬间预算可能不可申请
8. 驳回重提沿用原预算号时仍以当前额度为准
9. 提交失败不清空申请信息
10. 提交失败不清空产品申请订单明细
11. 提交失败不清空附件
12. 页面提示不得误导为“显示额度即保证提交成功”

#### 8.1.4 导入重复 / 冲突 / 覆盖说明必须覆盖的点
至少覆盖以下 10 点：

1. 同一店编多行归并
2. 同店编不同 SKU 保留多条产品
3. 同店编同 SKU 多行输入的当前口径
4. 不同店编同 SKU 分属不同明细
5. 导入当前是覆盖还是追加
6. 覆盖导入触发确认条件
7. 覆盖导入后折叠状态清空
8. 覆盖导入后金额重算
9. 软警告允许导入但提交仍阻断
10. 导入后手工修改以当前页面值为准

## 10. 执行顺序建议

### 第一步：优先修改创建页规则文档
原因：

1. 本轮一半以上新增说明与创建页直接相关
2. 预算并发、导入冲突、承接边界应先在创建页规则中定清
3. 创建页文档在 `CR-20260709-001` 后刚完成补强，连续补写成本最低

### 第二步：再修改详情页规则文档
原因：

1. 历史快照与实时主数据口径应在详情页统一收口
2. 详情页承接边界与现行展示模型应直接写入规则明细
3. 避免散写到 README 或任务说明里，造成入口分散

### 第三步：轻量回写 AI 开发任务说明
原因：

1. 任务说明只做承接协作约束补充
2. 不应替代规则文档本体

### 第四步：输出开发总结与日报
原因：

1. 本轮是规则文档治理型 CR
2. 必须形成完整闭环，方便后续继续承接

## 11. 本轮价值说明

本轮之所以值得做，不是因为页面主功能没做，而是因为当前项目已进入一个新的治理阶段：

> **页面主结构和主展示模型已经基本收口，接下来真正会影响后续开发、联调、验收效率的，是“承接层边界没写透”和“解释口径没有制度化”。**

如果不补本轮，后续最容易出现的问题不是“页面做不出来”，而是：

1. 正式工程同事不知道哪些规则必须沿用
2. AI 误把参考实现规则当成可随意替换的临时方案
3. 测试对“历史值 / 当前值”口径理解不一致
4. 产品与开发对“页面有额度但提交失败”的理解不一致
5. 导入链路在重复、冲突、覆盖场景下继续反复争议
6. 现有规则文档越来越多，但真正最关键的承接层说明仍然缺失

本轮的价值就是：

- 把“最容易被误解”的规则提前写进现有正式文档
- 不增加文档入口数量
- 不让开发和 AI 再靠猜测理解边界
