---

## 2. 变更背景

截至 `2026-07-10`，产品申请工单 App 参考实现已进入收尾阶段。当前主要风险已不再是“缺少文档”，而是：

1. **文档数量较多，主次层级不够直观**
2. **中英混合命名并存，不利于目录可读性与统一检索**
3. **主文档、页面规则文档、辅助文档、协作文档之间的阅读入口不够清晰**
4. **个别体量较小、内容边界较弱的文档单独存在，增加了阅读噪音**
5. **目录 README 仍偏向历史堆叠描述，尚未形成收尾阶段“主干清晰”的产品文档入口**

结合当前阶段目标，需先完成第一轮文档治理收口：  
**先把文档体系整理清楚，再进入第二轮重点文档内容合并、可读性与完整性优化。**

---

## 3. 本轮变更目标

本轮目标不是重写业务规则，而是完成以下 5 项治理动作：

1. **产品目录保留文档统一中文命名**
2. **明确主文档 / 页面规则文档 / 辅助文档 / 协作文档层级**
3. **新建一份产品文档总览入口文档**
4. **对小而散、重复度高的文档做低风险合并或降级处理**
5. **同步修正 README、正文交叉引用、版本说明中的旧文件名引用**

---

## 4. 固定前提与不可变口径

以下前提在本轮执行中继续生效，**不得回退、不得重定义、不得借治理之名改业务口径**：

### 4.1 固定工程与范围
- 主工程固定：`vue-reference/`
- React 归档：`legacy/react-reference/`
- 当前不以 React 作为实现依据

### 4.2 当前正式业务对象固定
- 产品申请工单
- 产品申请预算核销

### 4.3 必须继续保留的重点文档类型
以下文档类型属于本轮明确保留对象，不得误删、误并：

1. **PRD 主文档**
2. **页面规则明细文档**
3. **横切业务规则文档**
4. **审批承接边界文档**

### 4.4 不得回退的既有收口结论
- 发起人与审批人统一来源于“产品申请审批流程管理”
- 预算状态只有：草稿 / 已生效 / 已作废 / 已到期
- `applicationAvailable=false`：预算核销不可用，前端禁选
- `isAbnormal`：仅表示组织异常提示，不阻断使用
- 工单结构固定：工单头 + 多个产品申请订单明细 + 每个明细对应一个订单归属专卖店 + 多条 SKU/产品明细
- 用户术语统一：产品申请订单明细 / 产品 / 添加产品 / 添加产品申请订单
- 同店编多明细审批通过后正式订单归并规则保持不变
- 审批超时自动驳回、组织变更场景、财务通知与系统驳回边界、4 类驳回来源分类等既有收口结论保持不变
