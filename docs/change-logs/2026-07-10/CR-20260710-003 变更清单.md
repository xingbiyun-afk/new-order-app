原因如下：

1. 该类文档与“开发总结”高度重合

3. 与 0710 前两轮“文档收敛、结构瘦身、降噪”的目标相冲突

因此，0710 当日的正式总结统一通过开发总结承载。

### 4.2 关于承接清单的结论
本轮不新增《正式工程承接清单》。  
原因如下：

1. 正式工程承接清单不仅包含文档范围，还应覆盖代码范围、接口范围、联调范围、未纳入范围等内容

3. 更合理的时机应为：代码范围稳定后，再统一产出正式工程承接清单

因此，本轮只在开发总结与日报中记录该结论，不形成单独承接清单文档。

### 5.3 文档口径一致性复核

对当前产品文档做一轮轻量但系统性的口径复核，重点包括以下项目。

#### 5.3.1 旧概念残留排查
必须排查是否仍残留以下旧概念，一旦发现应清理或改正：

- 预算责任人管理
- 主责 / 备责
- 责任人异常影响预算核销
- 待调整预算状态
- 预算锁定 / 锁定状态 / 预算被锁定

#### 5.3.2 固定业务口径一致性排查
必须确认以下固定口径未被破坏：

- 发起人与审批人统一来源于“产品申请审批流程管理”
- 预算状态只有：草稿 / 已生效 / 已作废 / 已到期
- `applicationAvailable=false`：预算核销不可用，前端禁选
- `isAbnormal`：仅表示组织异常提示，不阻断使用
- 工单结构固定：工单头 + 多个产品申请订单明细 + 每个明细 = 一个订单归属专卖店 + 多条 SKU/产品明细
- 同一店编预算核销产品填写在一个明细
- 同店编多明细审批通过后生成正式订单时归并规则保持不变

#### 5.3.3 用户术语一致性排查
重点确认以下术语统一：

- 产品申请订单明细
- 产品
- 添加产品
- 添加产品申请订单

若存在旧叫法、混用叫法、临时叫法，应统一修订。

#### 5.3.4 金额展示口径排查
必须确认以下金额口径未回退：

- 剩余额度（净额）
- 预算总额（净额）
- 本次申请额度（净额）

禁止回退为：

- 可用金额
- 总额
- 申请金额（旧口径，如脱离“净额”语义）

同时检查：

- 金额是否统一要求千分符
- 是否统一保留两位小数
- 金额计算公式表述是否一致
- 折扣口径是否仍明确为“非常量”

#### 5.3.5 附件规则口径排查
确认以下附件规则表达一致：

- 附件必传
- 支持 JPG、PNG、GIF、Word、PDF、PPT、Excel
- 单文件 ≤ 20MB
- 最多 10 个附件
- 发起页支持上传 / 预览 / 删除
- 详情页支持预览 / 下载

#### 5.3.6 文档职责边界排查
检查以下边界是否仍清晰：

1. PRD 不过度重复页面执行细则
2. 页面规则文档仍是页面直接执行依据
3. 页面结构草图说明不重复按钮级交互与校验细则
4. 字段清单不重复业务流程与审批流规则
5. 审批承接说明不扩写后台完整页面规则体系
6. 预算核销规则文档负责专项规则，不替代 PRD 或页面规则文档

#### 5.3.7 文档间引用关系排查
检查：

1. 文首“文档定位”是否与实际内容匹配
2. 上下游引用关系是否正确
3. 是否存在引用旧文件名、旧路径、已移除文档名的情况
4. `README.md`、`产品申请工单文档总览.md`、`产品申请工单AI开发任务说明.md` 是否已同步当前正式结构

### 5.4 代码轻量巡检

对 `vue-reference/` 做一轮轻量代码巡检，目标是确认当前实现未与正式文档形成明显冲突。  
注意：本轮是巡检，不是代码主整改专项。

#### 5.4.1 巡检范围
建议巡检以下内容：

1. 页面目录与路由命名
2. 创建页相关实现
3. 预算选择页相关实现
4. 详情页相关实现
5. 我的页相关实现
6. 与产品申请工单相关的 mock / store / types / utils
7. 文案、状态、字段命名、注释等低风险层面

#### 5.4.2 巡检重点
重点检查以下问题：

##### A. 页面与文档命名一致性
- 页面标题是否仍使用旧口径
- 组件命名/注释是否存在旧术语
- 路由标题/菜单标题是否与正式文档明显冲突

##### B. 固定规则承接情况
- `applicationAvailable=false` 是否前端禁选
- `isAbnormal` 是否仅做提示不阻断
- 附件数量上限是否已体现
- 附件规则文案是否存在明显错误
- 金额展示是否符合净额口径
- 金额格式是否统一两位小数、千分位

##### C. 状态与驳回规则承接情况
- 主状态是否仍保持既定口径
- 是否误把 4 类驳回来源实现成 4 个主状态
- 详情页是否能区分驳回原因/来源与主状态
- 是否存在“财务通知即驳回结果”的错误实现或文案

##### D. 详情页阅读路径承接情况
- 是否符合已验收结论：
  - 展示优化
  - 默认展开策略
  - 阅读路径优化
  - 视觉层级减负
- 是否误新增复杂区块或重复区块

##### E. 旧概念残留排查
代码内是否仍残留以下内容：
- budget owner / 责任人
- lock / locked / 锁定
- adjust status / 待调整
- 主责 / 备责
- 影响预算核销的异常阻断语义

#### 5.4.3 巡检结果处理原则
1. 若无冲突，则在开发总结与日报中明确“代码轻量巡检通过，无需代码修改”
2. 若发现少量低风险问题，则允许顺手修正
3. 若发现结构性问题，不在本轮扩大处理，只记录问题并说明后续建议
4. 禁止因巡检扩展为大面积代码重构
