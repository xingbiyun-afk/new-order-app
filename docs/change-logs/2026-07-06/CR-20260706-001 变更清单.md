# CR-20260706-001 产品申请工单详情页结构与字段层级优化

## 1. 变更背景

截至当前，以下内容已完成并收口：

- `CR-20260703-001 产品申请工单选择链路与驳回后重新发起体验优化`
- `CR-20260703-002 产品申请工单创建链路收尾与样式口径统一补充`
- `CR-20260703-003 产品申请工单详情页驳回态与重新发起链路优化`

其中，`CR-20260703-003` 已完成详情页驳回 Banner、重新发起入口差异化、冻结期确认弹窗、金额口径统一补丁、产品明细汇总折叠、mock 场景补强等首轮优化。

但在当前版本下，详情页仍存在以下问题：

1. 页面区块虽已较完整，但整体阅读路径仍不够稳定，用户进入详情页后，首屏与次级信息区的主次关系不够明确；
2. 基础信息、发起信息、预算与申请信息、产品申请订单明细、附件、审批流、订单结果等区块虽然都已存在，但字段优先级和分组关系仍有优化空间；
3. 不同状态（处理中 / 已驳回 / 已结束）下，页面重点区域虽有差异，但尚未形成统一且稳定的结构承接方案；
4. 产品申请订单明细区已经具备折叠与汇总能力，但仍需进一步明确“卡片头信息”“展开后字段”“金额信息”“结果信息”的层级；
5. 审批流区与订单结果区当前已能展示信息，但在整体结构中仍偏“后加功能块”，尚未完全融入页面主结构；
6. 后续若要正式补建《产品申请工单详情页面规则明细.md》，需要先将详情页结构与字段层级收敛到较稳定状态。

因此，本轮需在 `CR-20260703-003` 的基础上，进一步收口详情页结构与字段层级问题，并顺带统一状态表达、金额信息层级、附件区信息密度与 mock 场景验证方式，为下一轮“审批流 / 功能订单 / 关联订单展示优化”以及后续详情页规则文档建档打基础。

---

## 2. 变更目标

本轮目标如下：

1. 明确详情页整体区块排序与阅读路径；
2. 明确不同状态下（处理中 / 已驳回 / 已结束）的首屏重点信息与次级信息；
3. 优化基础信息区、发起信息区、预算与申请信息区、产品申请订单明细区、附件区、审批流区、订单结果区之间的字段层级；
4. 统一详情页中的状态提示、失败提示、金额主次关系与文案表达；
5. 通过 mock 场景验证详情页在复杂状态、多明细、多附件、多结果下的可读性；
6. 为后续“审批流 / 功能订单 / 关联订单展示优化”与“详情页规则明细文档”输出稳定页面基础。

## 3. 变更范围

本轮变更范围包括：

1. 产品申请工单详情页整体页面结构；
2. 基础信息区、发起信息区、预算与申请信息区；
3. 产品申请订单明细区；
4. 附件区；
5. 审批流区；
6. 订单结果区；
7. 不同状态下的区块展示优先级与字段层级；
8. 详情页相关 mock 场景和辅助数据承接；
9. 与上述内容相关的正式文档回写。

## 4. 详情页整体结构重排要求

### 4.1 本轮结构目标
详情页整体结构需稳定为“先看状态和结论，再看基础背景，再看明细与过程，再看结果与附件”的阅读顺序。

### 4.2 建议区块顺序
本轮要求详情页按以下优先顺序组织：

1. **顶部状态区**
2. **驳回信息区**（仅已驳回时展示）
3. **基础信息区**
4. **发起信息区**
5. **预算与申请信息区**
6. **产品申请订单明细区**
7. **附件区**
8. **审批流区**
9. **订单结果区**
10. **操作区**

### 4.3 结构要求
1. 不同区块之间需有明确标题与分隔，不允许仅靠留白弱区分；
2. 同一层级区块标题样式应统一；
3. 页面首屏不应堆叠过多次级字段；
4. 若某区块本状态下无有效信息，应允许不展示或弱化展示，不做空壳占位。

### 4.4 本轮验收口径
验收时需重点判断：
- 用户进入详情页 3~5 秒内，能否快速理解“当前状态 / 原因 / 下一步动作”；
- 页面是否仍存在“所有区块都重要、阅读顺序不清”的问题。

---

## 5. 不同状态下的区块优先级要求

### 5.1 处理中
处理中场景下，页面优先级应为：

1. 当前状态
2. 当前审批进度 / 待处理信息
3. 发起背景与申请信息
4. 产品申请订单明细
5. 附件
6. 审批流过程
7. 订单结果（若尚未生成，可弱化或后置）

### 5.2 已驳回
已驳回场景下，页面优先级应为：

1. 驳回信息（原因 / 时间 / 驳回人 / 备注）
2. 是否允许重新发起 / 如何重新发起
3. 发起背景与预算申请信息
4. 产品申请订单明细
5. 附件
6. 审批流
7. 历史结果信息（如有）

### 5.3 已结束
已结束场景下，页面优先级应为：

1. 最终状态
2. 订单结果 / 分组结果 / 失败原因
3. 发起背景与申请信息
4. 产品申请订单明细
5. 审批流
6. 附件

### 5.4 验收要求
1. 不同状态下，首屏重点区块必须有明显差异；
2. 不允许所有状态只换一个状态标签、其余结构完全相同；
3. 页面需体现“状态驱动的信息主次变化”。

---

## 6. 基础信息区、发起信息区与预算/申请信息区字段层级要求

### 6.1 基础信息区
建议展示字段包括但不限于：

- 工单号
- 当前状态
- 申请人
- 申请组织
- 创建时间

### 6.2 发起信息区
建议展示字段包括但不限于：

- 申请类型
- 申请理由

### 6.3 预算与申请信息区
建议展示字段包括但不限于：

- 预算号
- 预算归属组织
- 预算使用范围
- 预算总额（净额）
- 本次申请额度（净额）

### 6.4 字段层级要求
1. 基础信息区优先放“识别工单身份”的字段；
2. 发起信息区优先放“为什么发起”的字段；
3. 预算与申请信息区优先放“本次申请与预算关系”的字段；
4. 不允许同一个字段在多个区块重复展示；
5. 若某字段在首屏已强化展示，次级区块中应避免再次高强度重复。

### 6.5 金额口径要求
1. 所有金额统一使用千分符 + 两位小数；
2. “预算总额（净额）”为辅助信息，可弱化；
3. “本次申请额度（净额）”为主信息，可重点突出；
4. 不得回退为：
   - `可用金额`
   - `总额`
   - 无千分符
   - 位数不统一

---

## 7. 产品申请订单明细区结构要求

### 7.1 区块目标
详情页中的“产品申请订单明细”必须做到：

- 先让用户知道这张工单包含多少笔明细、多少个专卖店、多少个产品；
- 再让用户按组查看具体内容；
- 不在首屏一次性平铺全部产品明细。

### 7.2 汇总区要求
汇总区建议至少包含：

- 产品申请订单明细数量
- 涉及专卖店数量
- 产品数量

### 7.3 卡片头要求
每个“产品申请订单明细”卡片头建议至少包含：

- 明细编号
- 订单归属专卖店
- 产品数量
- 分组金额

### 7.4 展开区要求
展开后建议展示：

- 产品编号 / 产品名称
- 数量
- 单价 / JDE价格（如当前详情页已定义展示）
- 产品金额
- 必要的打折标签或说明（如保留）

### 7.5 结构要求
1. 默认折叠可保留；
2. 折叠态必须能提供足够摘要信息；
3. 多明细场景下，卡片之间必须具备可快速区分的视觉特征；
4. 与创建页的“产品申请订单明细”命名与编号风格尽量保持一致。

---

## 8. 审批流区与订单结果区的层级收敛要求

> 说明：本轮不是“审批流 / 功能订单 / 关联订单展示优化”的主 CR，但需要提前做结构收敛准备。

### 8.1 审批流区
本轮要求：

1. 审批流区继续保留当前时间线承接；
2. 节点状态区分方式要稳定，不在本轮大幅重做；
3. 审批流区在页面中的位置需固定，不再反复前后调整；
4. 已驳回场景下，审批流区不能抢过驳回信息区；
5. 处理中场景下，审批流区应比已结束场景更重要。

### 8.2 订单结果区
本轮要求：

1. 订单结果区保留当前结果展示方向；
2. 已结束场景下其层级应明显提升；
3. 失败原因需与对应结果保持足够接近，避免用户来回对照；
4. 本轮不展开复杂的“功能订单 / 关联订单 / 分组结果完全重构”，但要为下一轮留出稳定区块位置。

---

## 9. 状态提示条、失败提示与异常提示优化要求

### 9.1 本轮目标
统一详情页中的提示表达，避免出现不同区块各自使用不同提示风格、文案强度不一致的问题。

### 9.2 提示类型
本轮需重点统一以下提示：

1. 驳回提示
2. 不可继续基于原单重提提示
3. 结果失败提示
4. 异常提示（如当前详情页已有）

### 9.3 要求
1. 不同提示类型需有统一的颜色、层级和区块风格；
2. 提示语需尽量业务化、用户可理解；
3. 避免仅展示技术判断结果而不解释含义；
4. 驳回提示与失败提示不得混用同一层级，导致用户误解。

---

## 10. 附件区轻量优化要求

### 10.1 本轮目标
附件区不做系统级重构，但要进一步提升可读性与使用清晰度。

### 10.2 本轮要求
1. 文件名称过长时需保证可读性；
2. 文件类型图标、文件大小、文件名称之间层级清楚；
3. 预览 / 下载入口表达明确；
4. 多附件场景下排布不应过乱；
5. 附件区不应在已驳回场景中抢过驳回信息区。

---

## 11. Mock 数据持续补强与结构验证要求

### 11.1 本轮目标
本轮所有结构调整必须在多个场景下验证，不允许只在一个 mock 页面下调通后即判定完成。

### 11.2 必验场景
至少需覆盖以下场景：

1. 处理中（多审批节点）
2. 已驳回（冻结期）
3. 已驳回（已到期）
4. 已驳回（非冻结期）
5. 已结束（部分成功、部分失败）
6. 多产品申请订单明细
7. 多附件
8. 审批流较完整
9. 订单结果较复杂

### 11.3 要求
1. 调整后的页面需在上述场景中检查结构是否失衡；
2. 若某场景暴露明显结构问题，本轮需同步修正；
3. 必须确保详情页不是只适配单一 happy path 场景。

## 14. 开发与验收要点

### 14.1 结构层
1. 页面整体阅读路径必须更清晰；
2. 区块顺序需体现主次关系；
3. 不同状态下的首屏重点区块必须有差异。

### 14.2 字段层
1. 同类字段应聚合；
2. 重点字段应突出；
3. 非重点字段应弱化；
4. 不得引入新业务概念或旧口径回退。

### 14.3 产品申请订单明细区
1. 汇总区、卡片头、展开区三层结构必须清晰；
2. 默认折叠策略需兼顾效率与可读性；
3. 多明细场景下需仍可快速区分。

### 14.4 状态与金额
1. 状态提示层级必须统一；
2. 金额主次必须服务当前状态；
3. 文案口径不得回退。

### 14.5 mock 数据
1. 至少完成必验场景检查；
2. 不允许只对单一场景优化；
3. mock 数据应能支撑结构稳定性验收。
