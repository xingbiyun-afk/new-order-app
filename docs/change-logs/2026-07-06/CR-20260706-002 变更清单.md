# CR-20260706-002 产品申请工单详情页审批流、订单结果展示优化与白功能项收口

## 1. 变更背景

`CR-20260706-001 产品申请工单详情页结构与字段层级优化` 已完成详情页主结构、字段分区、金额层级、产品申请订单明细区、附件区、提示条等基础收口，详情页整体阅读路径已相对稳定。

在此基础上，当前详情页仍存在以下待收口问题：

1. 审批流区虽已能展示基础时间线，但节点字段层级、当前节点识别、审批意见贴近展示、多节点可读性仍需进一步优化；
2. 订单结果区已有初步承接，但“正式订单结果”“失败原因”“预占库存订单（功能订单）信息”之间的主次关系还不够稳定；
3. 当前尚未把“预占库存订单（功能订单）”与最终正式订单的展示边界完全说清，易导致详情页结果区继续膨胀；
4. `CR-20260706-001` 之后，详情页下一优先级已明确为“审批流 / 功能订单 / 关联订单展示优化”，需尽快形成可执行、可验收的正式变更清单；
5. 项目当前仍存在 `uuid` moderate 依赖安全遗留，虽不属于主业务功能，但需要纳入白功能清单讨论范围，形成风险复核与后续建议结论。

此外，当前业务规则已进一步明确：

- 工单发起后，系统会根据工单内申请的产品 SKU 和数量生成对应的**预占库存订单（功能订单）**，用于占用库存；
- 工单审批通过后，预占库存订单取消，随后生成对应正式订单：
  - 客户层级 = 6 → **产品申请表订单**
  - 客户层级 = 0-4 → **内部申请表订单**
- 工单中途驳回后，对应预占库存订单同样取消；
- 因此，预占库存订单的最终结果恒为“取消”；
- 详情页中无需将预占库存订单作为复杂状态结果区块承接，本轮只需展示其**订单编号**；
- 一般口径下，**每一笔产品申请订单明细对应一个预占库存订单编号**。

因此，本轮需在已稳定的详情页主结构基础上，进一步收口审批流区、订单结果区及预占库存订单编号展示方式，并同步完成白功能项结论整理，为后续详情页正式规则文档建档打基础。

## 2. 变更目标

本轮目标如下：

1. 收口详情页审批流区的节点信息结构与阅读层级；
2. 收口详情页订单结果区的正式订单结果展示方式、失败原因贴近展示方式及明细 / 归并关系表达；
3. 明确预占库存订单（功能订单）在详情页中的展示边界：**只展示订单编号，不展开复杂状态过程**；
4. 通过 mock 场景验证复杂审批流、多结果、部分成功 / 部分失败、归并生成等场景下的结构稳定性；
5. 将 `uuid` moderate 风险纳入白功能项讨论范围，形成风险复核、影响说明和后续建议，不在本轮强制修复；
6. 同步将本轮结构与规则结论回写到正式产品、结构、任务类文档，保证后续 AI / 开发执行口径一致；
7. 为后续《产品申请工单详情页面规则明细》正式建档提供稳定前置结构基础。

## 4. 审批流区优化要求

### 4.1 本轮目标

审批流区需做到“让用户看得清当前走到哪一步、谁处理过、结果是什么、意见写了什么”，但不做流程系统级重构。

### 4.2 节点信息要求

每个审批节点建议至少展示：

- 节点名称
- 审批人
- 审批结果
- 审批时间
- 审批意见 / 备注（如有）

如当前节点尚未处理，可展示：

- 当前待处理节点
- 当前待处理人 / 待审批角色
- 当前节点状态说明

### 4.3 状态驱动要求

1. **处理中**
   - 当前节点信息优先；
   - 与顶部状态区 / 当前审批进度卡片形成承接；
   - 待处理节点与已处理节点区分清楚；
   - 不应让用户进入审批流区后仍无法快速判断“卡在哪一步”。

2. **已驳回**
   - 驳回节点应在审批流中可快速识别；
   - 驳回意见与对应驳回节点贴近展示；
   - 审批流区不能抢过顶部驳回信息区；
   - 不得把驳回原因拆散到多个位置导致用户来回对照。

3. **已结束**
   - 审批流区以过程追溯为主；
   - 层级低于订单结果区；
   - 仍需保证节点信息完整可读；
   - 不因已结束场景而把审批流压缩成不可阅读的简略占位。

### 4.4 结构与视觉要求

1. 当前节点、已完成节点、已驳回节点、待处理节点需有稳定区分方式；
2. 多节点场景下，时间、结果、意见不可相互挤压；
3. 审批意见不应远离对应节点；
4. 同类节点信息的排版方式应统一；
5. 本轮不大改节点命名机制，不引入新的审批业务概念；
6. 不新增正式审批状态枚举，仍以展示字段承接。

### 4.5 本轮验收口径

验收时需重点判断：

- 用户能否快速识别当前审批节点；
- 多节点场景下审批流是否仍清晰可读；
- 驳回意见是否贴近驳回节点；
- 审批流区是否仍然出现“有信息但阅读成本高”的问题。

## 5. 订单结果区优化要求

### 5.1 本轮目标

订单结果区需做到“先让用户看清最终生成了什么正式订单，再让用户理解失败了什么、归并到了哪里、对应哪笔明细”，不再把所有结果信息平铺堆叠。

### 5.2 结果主次关系

本轮结果区主次关系要求如下：

1. **正式订单结果**是主信息；
2. **失败原因**需贴近对应正式订单结果；
3. **预占库存订单编号**是辅助信息；
4. 不将预占库存订单作为复杂结果区块独立放大展示；
5. 不把“过程性订单信息”和“最终业务结果”混在同一层级表达。

### 5.3 正式订单结果展示要求

每笔结果建议至少展示：

- 对应的产品申请订单明细编号
- 订单归属专卖店
- 正式订单结果状态
- 正式订单类型
- 正式订单编号（如成功生成）
- 失败原因（如失败）
- 必要的归并说明（如该结果由多笔明细归并生成）

### 5.4 正式订单类型要求

需明确按客户层级承接：

- 客户层级 = 6 → **产品申请表订单**
- 客户层级 = 0-4 → **内部申请表订单**

如存在归并生成，需清楚表达明细与正式订单之间的对应关系，不允许只展示一个订单号而不说明由哪些明细归并而来。

### 5.5 失败信息要求

1. 失败原因必须贴近对应结果；
2. 不允许把失败原因统一堆在区块底部让用户来回对照；
3. 部分成功、部分失败场景下，成功结果与失败结果需能快速区分；
4. 失败原因文案应尽量业务化、可理解，不只给技术判断结果。

### 5.6 已结束场景的层级要求

1. 已结束场景下，订单结果区层级需明显高于审批流区；
2. 如存在“全部成功”“部分成功”“全部失败”等不同结果状态，页面需有清晰区分；
3. 已结束场景首屏应优先承接最终结果，而不是把结果埋到页面后部。

### 5.7 本轮验收口径

验收时需重点判断：

- 用户能否快速区分最终正式订单结果与过程性预占库存订单信息；
- 失败原因是否贴近对应正式订单结果；
- 多结果 / 归并生成场景下，页面是否仍具有可读性；
- 结果区是否仍存在“信息全有，但主次不清”的问题。

## 6. 预占库存订单（功能订单）展示边界要求

### 6.1 业务规则承接

本轮需明确以下规则并在页面中正确承接：

1. 工单发起后，系统按工单内申请的产品 SKU 和数量生成预占库存订单，用于占用库存；
2. 工单审批通过后，预占库存订单取消，再生成正式订单；
3. 工单中途驳回后，预占库存订单同样取消；
4. 因此，预占库存订单最终结果恒为“取消”。

### 6.2 页面定位要求

基于上述规则，预占库存订单在详情页中的定位应明确为：

- **库存占用过程信息**
- **不是最终业务结果主信息**
- **不作为复杂结果区块展开**
- **本轮仅承接订单编号展示**

### 6.3 页面展示要求

详情页中对预占库存订单的展示要求为：

1. **只展示预占库存订单编号**；
2. 一般按**每一笔产品申请订单明细对应一个预占库存订单编号**承接；
3. 展示层级低于正式订单结果；
4. 可作为结果卡片中的辅助字段展示；
5. 如页面结构更合适，也可贴近对应的产品申请订单明细结果展示；
6. 本轮不展开其状态流转、取消过程、取消原因、详情跳转等复杂展示。

### 6.4 文案与层级要求

1. 预占库存订单相关文案需避免误导用户将其理解为最终订单；
2. 不得将预占库存订单编号以大标题、大色块等方式放大为主视觉信息；
3. 如与正式订单编号同时展示，需有明确字段名称区分；
4. 不允许“订单编号”字段不加说明同时混放，导致用户无法判断编号类型。

### 6.5 本轮验收口径

验收时需重点判断：

- 页面是否把预占库存订单错误放大为复杂主结果；
- 用户是否能快速分清“预占库存订单编号”和“正式订单结果”；
- 页面是否仍以正式订单结果为主要结论承接。

## 7. 结果分组与归并展示要求

### 7.1 结果阅读入口

页面仍以**产品申请订单明细维度**作为结果阅读入口。

也就是说，用户应能从每一笔产品申请订单明细出发，理解：

- 对应的预占库存订单编号是什么；
- 最终生成了什么正式订单；
- 是否生成失败；
- 如归并生成，归并到了哪个正式订单。

### 7.2 归并展示要求

如多个明细在审批通过后归并生成同一正式订单，需在结果区明确表达：

1. 哪些产品申请订单明细归并到同一正式订单；
2. 对应正式订单类型与正式订单编号；
3. 如其中部分结果失败，失败原因仍需贴近对应结果；
4. 不得只展示一个订单号，让用户自行猜测对应哪些明细。

### 7.3 页面结构要求

1. 结果区需兼容“单明细单结果”和“多明细归并单结果”两种场景；
2. 归并说明不可喧宾夺主，但必须足够清楚；
3. 若某正式订单由多个明细归并生成，可采用结果卡片内列示明细编号的方式承接；
4. 本轮以移动端可读性优先，不追求一次性做成复杂表格结构。

### 7.4 本轮不纳入范围

本轮不要求把结果区重做成后台订单查询式的复杂表格结构。

## 8. 状态驱动的审批流区与结果区优先级要求

### 8.1 处理中

处理中场景下，页面重点应为：

1. 当前状态
2. 当前审批进度
3. 审批流中的当前节点
4. 发起背景与申请信息
5. 产品申请订单明细
6. 附件
7. 订单结果（如尚未生成，可弱化或后置）

### 8.2 已驳回

已驳回场景下，页面重点应为：

1. 驳回信息
2. 是否允许重新发起 / 如何重新发起
3. 发起背景与预算申请信息
4. 产品申请订单明细
5. 附件
6. 审批流（强调驳回节点）
7. 历史结果信息（如有）
8. 预占库存订单编号（如展示，不得抢主视线）

### 8.3 已结束

已结束场景下，页面重点应为：

1. 最终状态
2. 正式订单结果
3. 失败原因
4. 归并结果说明
5. 审批流
6. 附件
7. 预占库存订单编号（辅助信息）

### 8.4 验收要求

1. 不同状态下，审批流区与结果区的权重需有明显差异；
2. 不允许所有状态下区块层级完全一致；
3. 页面需体现“状态驱动的信息主次变化”。

## 10. 白功能项：依赖安全风险复核与结论要求

### 10.1 背景

当前项目仍存在 `uuid` moderate 告警，来源于 `exceljs@4.4.0` 的传递依赖，非当前业务直接依赖。

### 10.2 本轮目标

本轮需将该问题纳入白功能项讨论范围，并形成结论，至少包括：

1. 当前风险来源说明；
2. 当前影响范围说明；
3. 不采用 `npm audit fix --force` 的原因；
4. 对当前 Excel 导入解析与动态加载能力的潜在影响说明；
5. 后续建议路径。

### 10.3 固定口径

本轮必须明确以下固定口径：

1. **不采用 `npm audit fix --force`**；
2. 不为消除告警而直接破坏现有 Excel 导入能力；
3. 本轮以“风险复核 + 结论整理”为主，不强制要求本轮完成升级替换；
4. 后续可继续关注：
   - `exceljs` 上游修复
   - `uuid` 升级可行性专项评估
   - SheetJS 社区版
   - PapaParse 纯 CSV 方案等替代路径

### 10.4 本轮交付要求

本轮至少需在以下任一载体中形成明确记录：

1. 变更清单；
2. 开发总结；
3. 项目日报；
4. 必要时补充轻量技术评估记录。

### 10.5 本轮验收口径

验收时需重点判断：

- 是否明确说明 `uuid` moderate 来源于 `exceljs` 传递依赖；
- 是否明确说明本轮不强制修复；
- 是否明确说明不采用 `npm audit fix --force`；
- 是否形成了后续跟踪建议，而不是只停留在“有风险”层面。

1. 必须覆盖多节点、多结果、归并生成场景；
2. 不允许只做单场景验证；
3. Mock 数据应能支撑结构稳定性验收。

### 12.5 白功能项

1. 必须形成风险结论；
2. 不得写成“本轮已强制修复 uuid moderate”；
3. 不得使用 `npm audit fix --force` 作为本轮方案。
