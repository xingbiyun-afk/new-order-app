## 2. 变更背景

截至 `2026-07-07`，详情页相关工作已连续完成并收口以下变更：

- `CR-20260706-001 产品申请工单详情页结构与字段层级优化`
- `CR-20260706-002 产品申请工单详情页审批流、订单结果展示优化与白功能项收口`
- `CR-20260706-003 产品申请工单详情页面规则明细建档、Mock 数据纠偏与轻量展示收口`
- `CR-20260706-004 订单结果区专卖店变更、失败原因类型、备注显示`
- `CR-20260707-001 详情页现行规则口径统一、协作文档收口与说明性修正`

其中，详情页整体结构、审批流、订单结果区主次关系、预占订单展示边界、失败原因类型、专卖店变更、备注展示等方向均已基本收口。但在订单结果区“多次生成尝试 / 最终成功 / 最终失败 / 业务放弃”相关场景中，仍存在以下需要继续统一的问题：

### 2.1 当前已知问题

1. **重试历史的外层摘要与展开区边界还不够清晰**
   - 当前虽已支持“生成尝试历史”折叠，但外层卡片应该展示“最新结果摘要”，还是继续堆叠过往失败原因，规则尚不够明确；
   - 历史失败原因、业务备注、最新结果之间主次关系仍有歧义。

2. **最终结果已成功时，外层卡片仍可能继续显示失败提示**
   - 例如 `PA202407050001` 对应的成功记录中，Mock 数据已承接成功结果，但卡片外层仍显示“订单生成失败”提示，容易让用户误解当前结果仍为失败；
   - 该问题本质上是“历史失败过程信息”未与“最新结果状态”正确区分。

3. **多次失败后业务放弃场景的备注展示口径需要收口**
   - 当前业务明确：当订单多次生成失败后，若业务决定放弃，需要在**最新失败的订单草稿**中填写备注说明后，才可以删除草稿；
   - 因此在页面展示上，应以“最新失败记录对应备注说明”作为外层主承接内容，而不是在外层重复堆叠多条失败备注。

4. **订单结果状态术语仍存在历史表述混用**
   - 当前页面、Mock、说明文档中仍可能出现“已生成 / 生成成功”等阶段性表述；
   - 但根据系统现有订单状态定义，本工单场景下订单结果只需要承接以下 4 种状态：
     1. 已创建
     2. 客服已审核
     3. 财务已审核
     4. 草稿
   - 因此成功类结果状态口径需统一为“已创建”，不再使用“已生成”作为正式状态文案。

5. **订单结果区对系统真实链路的承接关系尚未在文档与 Mock 中完全体现**
   - 工单审批通过后，并非直接“生成正式订单”，而是先取消预占库存订单，再生成对应订单草稿，再自动提交成为正式订单；
   - 若自动提交失败，则会回退到草稿状态；
   - 已创建订单在客服/财务审核中被驳回时，原订单编号会被系统物理删除，并重新关联回原草稿 ID，回退到草稿状态；
   - 这些关键链路需要在订单结果区展示规则、Mock 数据和规则文档中统一体现。

6. **重试历史展开区排版存在可读性问题**
   - 当前长文本失败原因或备注集中排在最右侧时显得拥挤；
   - 需要在不重做整体结构的前提下进行轻量优化，使历史记录更易读。

因此，本轮不新增业务功能，不改动正式订单系统规则，而是对**详情页订单结果区在“重试历史 / 最终结果状态 / 草稿回退 / 业务放弃备注 / Mock 规则映射”方向上的展示承接与术语口径**进行统一收口，避免后续继续出现“最终状态已成功但页面仍按失败展示”“已创建与已生成混用”“草稿语义不清”“历史失败信息外溢”等问题。

## 3. 本轮变更目标

本轮目标共 5 项：

1. **收口订单结果区重试历史的展示边界**
   - 明确“外层卡片仅承接最新结果摘要、历史尝试记录默认折叠”的规则；
   - 明确一直失败、最终已创建、业务放弃等场景下，外层与折叠区分别展示什么内容。

2. **统一订单结果状态术语**
   - 明确本工单场景下订单结果只承接 4 种状态：
     1. 已创建
     2. 客服已审核
     3. 财务已审核
     4. 草稿
   - 成功类结果统一使用“已创建”，不再使用“已生成 / 生成成功”作为正式状态文案。

3. **让订单结果区展示与系统现有链路对齐**
   - 明确“审批通过 → 取消预占库存订单 → 生成草稿 → 自动提交 → 已创建 / 草稿回退”的链路承接；
   - 明确“已创建审核驳回后删除原订单号并回退草稿”的展示语义。

4. **修正关键 Mock 场景与页面摘要不一致问题**
   - 重点修正 `PA202407050001`、`PA202407060001` 等多次尝试场景；
   - 保证 Mock 数据、页面外层摘要、折叠区历史、备注口径一致。

5. **优化重试历史展开区的长文本排版**
   - 在不大改整体结构的前提下，对失败原因/备注信息做轻量排版调整；
   - 降低右侧拥挤、阅读顺序混乱的问题。

## 4. 固定执行口径

本轮执行必须严格遵守以下固定口径，不得回退：

### 4.1 正式业务对象
- 产品申请工单
- 产品申请预算核销

### 4.2 不得重新引入的旧概念
- 预算责任人管理
- 主责 / 备责
- 责任人异常影响预算核销
- 待调整预算状态
- 预算锁定 / 锁定状态 / 预算被锁定

### 4.3 详情页现行固定口径（本轮继续沿用）
1. 页面主状态固定承接：
   - 处理中
   - 已驳回
   - 已结束

2. 处理中场景不得展示正式订单结果，包括但不限于：
   - 产品申请表订单编号
   - 内部申请表订单编号
   - 正式订单结果卡片
   - `retryHistory`
   - 正式订单失败原因

3. 预占库存订单属于过程信息，不是主结果；

4. 处理中场景下，预占订单编号只在审批流“发起工单”节点展示；

5. 当前详情页展示与规则收口，以：
   - `docs/product/work-order-budget/产品申请工单详情页面规则明细.md`
   为现行直接执行依据。

### 4.4 订单结果状态固定口径

当前系统在本工单相关订单结果展示中，仅承接以下 4 种订单状态：

1. **已创建**
   - 含义：订单已从草稿提交成为正式订单。

2. **客服已审核**
   - 含义：订单已通过客服部门审核。

3. **财务已审核**
   - 含义：订单已通过财务部门审核（终审）。

4. **草稿**
   - 含义包括但不限于：
     1. 订单未成功从草稿提交为正式订单；
     2. 自动提交过程中触发系统现有校验逻辑，回退到草稿；
     3. 已创建订单在客服审核或财务审核中被驳回，原订单编号被系统物理删除后重新关联回原草稿 ID。

#### 执行要求
1. 本工单详情页订单结果区只考虑上述 4 种状态；
2. 其他订单状态在当前工单场景中不展示、不扩展；
3. 文档、Mock、页面展示、说明性文案中，如表达“订单已从草稿提交为正式订单”，统一使用：
   - **已创建**
4. 不再使用以下表述作为正式订单状态文案：
   - 已生成
   - 生成成功
   - 订单已生成
5. 如页面中存在“生成尝试历史”等过程语义表述，可保留其“尝试过程”含义，但最终订单结果状态展示必须以系统正式状态为准。

### 4.5 订单结果链路承接规则

工单审批通过后，订单结果区承接的系统链路规则如下：

1. 工单审批通过后，系统先取消关联的预占库存订单；
2. 然后调用系统现有接口，生成对应的订单草稿：
   - 产品申请表订单草稿；
   - 内部申请表订单草稿；
3. 草稿生成后，系统再自动提交草稿，尝试将其提交为正式订单；
4. 自动提交成功后，订单状态承接为：
   - **已创建**
5. 自动提交过程中如发生系统现有草稿提交流程已支持的校验问题，则订单回退为：
   - **草稿**
6. 已创建订单在后续客服审核或财务审核过程中如被驳回：
   1. 原订单编号会被系统物理删除；
   2. 重新关联回原草稿 ID；
   3. 订单状态回退为：
      - **草稿**
7. 回退为草稿后，由业务同事根据实际情况处理，包括但不限于：
   1. 编辑草稿后重新发起；
   2. 删除草稿；
   3. 新建草稿并关联工单号。

#### 展示承接要求
1. 详情页订单结果区展示时，应按上述链路理解“已创建 / 草稿”等状态语义；
2. “草稿”并不等同于“未生成任何记录”，而可能表示：
   - 草稿已生成但自动提交失败；
   - 正式订单审核驳回后删除原订单编号并回退草稿；
3. 若展示失败说明，应结合实际语义表达“当前仍为草稿 / 自动提交失败回草稿 / 审核驳回回草稿”，避免把所有失败都笼统写成“已生成失败”。

## 6. 逐文件修改要求

## 6.1 `vue-reference/src/views/work-order-budget/DetailView.vue`

### 修改目标
统一订单结果区对“重试历史 / 最新结果摘要 / 草稿回退 / 业务放弃备注”的展示边界，修正当前与 Mock 语义不一致的展示问题，并进行轻量排版优化。

### 必改要求

#### A. 重试历史默认折叠
1. 当某关联订单存在多次尝试记录时：
   - 默认折叠；
   - 外层仅显示“生成尝试历史（N次）”入口或等价轻量入口；
2. 展开后可查看每次尝试的大致情况，包括但不限于：
   - 尝试时间
   - 订单编号 / 草稿编号（如已有承接）
   - 当前尝试结果状态
   - 对应失败原因或备注

#### B. 外层卡片只承接最新结果摘要
1. 外层卡片不得继续堆叠全部历史失败原因；
2. 外层卡片展示内容应以“最新一次结果”为准；
3. 历史失败详情统一放入折叠区域承接。

#### C. 一直失败场景
当某关联订单截至当前最终仍未进入“已创建 / 客服已审核 / 财务已审核”状态，且最新结果仍为失败/草稿时：

1. 外层卡片可展示：
   - 当前状态：草稿 / 生成失败语义下的草稿说明（以最终状态语义为准）
   - 最新一次失败摘要
2. 历史失败记录的失败原因/备注：
   - 放在折叠区展示；
   - 不在外层重复堆叠全部历史信息。

#### D. 业务放弃场景
当某关联订单属于“多次失败后业务决定放弃”场景时：

1. 外层卡片仅展示：
   - 最新失败记录对应的业务备注说明；
2. 不再在外层重复展示旧失败记录的备注；
3. 历史尝试信息仍在折叠区中保留。

#### E. 最终成功场景
当某关联订单经历过失败重试，但**最新结果已进入已创建状态**时：

1. 外层状态必须显示为：
   - **已创建**
2. 外层不得继续显示：
   - 订单生成失败
   - 订单已失败
   - 历史失败原因作为当前主结果
3. 历史失败信息仅保留在折叠区中供查看。

#### F. 已创建后审核驳回回草稿场景
如 Mock 或现有数据承接到“已创建后审核驳回回草稿”的结果：

1. 外层状态按：
   - **草稿**
   承接；
2. 说明语义应体现为：
   - 审核驳回后回草稿
   或等价表达；
3. 不得误展示为“仍然保留原已创建订单号且仅状态失败”。

#### G. 长文本排版轻量优化
1. 展开区中的失败原因/备注，不应继续全部压在最右侧窄列；
2. 建议改为：
   - 时间独立一行；
   - 订单号 + 状态一行；
   - 失败原因 / 备注独立换行；
3. 保持轻量，不做整体结构重构。

### 禁止事项
- 禁止新增订单结果区新的业务能力；
- 禁止扩展新的订单状态；
- 禁止回退为“已生成 / 生成成功”正式状态口径；
- 禁止把历史失败原因重新全部堆回外层主卡片；
- 禁止重新引入预占订单作为订单结果区主内容。

### 验收要点
- 外层卡片与折叠区主次关系清晰；
- 最终成功时不再出现失败主提示误导；
- 业务放弃时外层只承接最新失败备注；
- 展开区长文本可读性改善；
- 页面状态口径与系统定义一致。

## 6.2 `vue-reference/src/mocks/index.ts`

### 修改目标
修正订单结果相关 Mock 数据，使其与页面最新展示规则及系统状态语义一致。

### 必改要求

#### A. 统一成功状态口径
1. 所有表达“订单已从草稿提交为正式订单”的场景，状态文案统一改为：
   - **已创建**
2. 不再继续使用：
   - 已生成
   - 生成成功
   - 订单已生成

#### B. 补齐草稿语义
对“失败 / 回退 / 审核驳回”相关场景，Mock 数据需能体现以下语义之一：

1. 自动提交失败回草稿；
2. 客服审核驳回回草稿；
3. 财务审核驳回回草稿；
4. 多次失败后业务决定放弃（最新失败草稿带业务备注）；

并确保页面可基于 Mock 明确区分“最新结果是草稿还是已创建”。

#### C. 重点修正场景
1. `PA202407050001`
   - 当前若最终结果为成功，需确保其最新外层语义承接为：
     - **已创建**
   - 历史失败记录保留在折叠区；
   - 外层不再继续展示“订单生成失败”主提示。

2. `PA202407060001`
   - 若场景为多次失败后业务放弃：
     - 最新失败记录需带最终备注说明；
     - 外层只承接最新失败说明；
     - 历史失败原因放入折叠区。

#### D. 生成尝试历史字段统一
1. 如已有 `retryHistory` / `attempts` / 相关重试结构，需统一语义；
2. 每条历史建议至少承接：
   - 尝试时间
   - 订单编号或草稿标识
   - 状态
   - 失败原因 / 备注（视场景）
3. “最新结果”与“历史记录”在数据语义上应能区分，不得让页面只能通过硬编码猜测。

### 禁止事项
- 禁止继续保留与当前系统状态口径冲突的成功状态文案；
- 禁止出现“最终已创建，但摘要仍像失败”的 Mock 组合；
