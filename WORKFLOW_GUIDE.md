# WORKFLOW_GUIDE.md — 工作流程指南

## 当前项目定位

`new-order-app` 当前不是正式生产交付仓，而是：

1. 产品申请工单 / 产品申请预算核销需求文档沉淀仓；
2. 基于 Vue 的移动端样式、交互、结构参考实现仓。

当前阶段允许直接在本仓库中开展 Vue 参考实现，用于需求验证、交互确认、页面走查和后续正式开发承接。

> **2026-07-01 工程结构更新**：根目录默认运行与构建入口已切换为 Vue（`npm run dev` → `vue-reference/`）；历史 React 工程已归档至 `legacy/react-reference/`，需显式使用 `npm run dev:react-legacy` 运行。

## 当前正式业务口径

- 当前正式对象为：`产品申请工单` / `产品申请预算核销`
- 不是：`预算申请工单`
- 不恢复以下旧概念：
  - 预算责任人管理
  - 主责 / 备责
  - 责任人异常影响预算核销
  - 待调整预算状态
- 发起人与审批人来源以：`产品申请审批流程管理` 为准
- App 当前参考实现与后续正式开发方向：`Vue`

## 开发启动机制

当前阶段采用“首轮开发任务书 + 后续变更清单”模式：

1. 首轮开发由 `docs/product/work-order-budget/产品申请工单AI开发任务说明.md` 启动；
2. AI 先按该任务书完成第一轮 Vue 参考实现开发；
3. 首轮开发完成后进入验收；
4. 在该任务书基础上新增的需求、修订、补充、纠偏，不重写新的主任务书，而通过 `docs/change-logs/` 下的变更清单持续记录。

## 文档产出要求

### 1. 开发任务说明
用于启动首轮开发，定义当前开发目标、页面范围与执行边界。

### 2. 变更清单
用于记录首轮任务书基础上的新增需求、修订、补充和调整。

目录建议：
- `docs/change-logs/YYYY-MM-DD/`（日期子目录）
- 历史平铺路径保留，不强制迁移

命名建议：
- `CR-YYYYMMDD-NNN 变更清单.md`

### 3. 开发总结
每轮开发完成后必须补充开发总结，记录：

- 本轮目标
- 实现页面
- 关键规则承接
- 遗留问题
- 后续建议

目录建议：
- `docs/development-summary/YYYY-MM-DD/`（日期子目录）
- 历史平铺路径保留，不强制迁移

> AI 后续生成新文件时，默认使用日期子目录，不再优先旧平铺路径。

### 4. 项目日报
当天结束后补充项目日报，记录：

- 今日工作概览
- 已完成工作
- 文档产出
- 当前问题
- 下一步计划

目录建议：
- `docs/daily-reports/`

## 文档更新强制规则

自 `2026-07-01` 起，项目内凡被修改的**需求类、规则类、结构类文档**，不得只修改正文，必须同步完成文档变更记录补充。

### 适用范围
包括但不限于：

- `docs/product/` 下的 PRD、任务说明、字段清单、页面结构说明、规则明细等文档
- `docs/change-logs/` 关联变更单所引用并被更新的需求文档
- 其他用于沉淀产品规则、页面结构、字段口径、业务模型的文档

### 强制要求
每次修改上述文档时，必须同步完成以下内容：

1. **补充更新日志**
2. **标注本轮变更单号**
3. **附参考文档清单**
4. 如文档有版本号管理，还必须同步保证以下信息一致：
   - 文档标题中的版本号
   - 正文中的版本说明
   - 更新日志中的版本变更记录
   - 开发总结中的引用版本
   - 项目日报中的引用版本

### 禁止事项
禁止以下行为：

- 只修改正文，不补更新日志
- 只改版本号，不补变更摘要
- 已发生规则变更，但不标注变更单号
- 已引用其他需求文档，但不补参考文档清单
- 文档标题版本号、更新日志版本号、开发总结引用版本号互相不一致

### 执行说明
后续 AI 或人工在更新文档时，均应默认执行本规则。  
若某次变更涉及多个需求/规则/结构类文档，则这些文档均需分别补充更新日志，不得只在变更单中记录，而不回写到正文文档中。

### 最小落地格式建议
文档末尾建议统一保留：

- `更新日志`
- `参考文档`

如文档有版本管理，标题建议采用以下格式：

- `文档名称（v0.x）`

更新日志建议至少包含：

- 日期
- 版本变化（如适用）
- 变更单号
- 变更类型
- 变更摘要
- 参考文档

## 标准开发流程

### 1. 需求理解
- 阅读产品文档（`docs/product/`）
- 阅读技术文档（`docs/technical/`）
- 阅读 AI 协作基线与工作流文档
- 明确本期范围和不纳入范围的功能

### 2. 任务启动
- 确认当前是否已有正式开发任务说明
- 若是首轮开发，先按开发任务说明执行
- 若是后续调整，新增变更清单后再执行

### 3. 分支创建
- 从 `main` 创建新分支
- 分支名遵循 `BRANCH_GUIDE.md` 规范

### 4. 开发实施
- 按 feature 目录组织代码
- 当前 App 参考实现以 Vue 为准
- 优先复用现有工单移动端交互风格
- 保持移动端优先的设计理念
- 不擅自扩展未确认业务规则

### 5. 构建验证
- 开发完成后执行 `npm run build`
- 确保构建无错误

### 6. 提交规范
- 提交信息格式：`{type}: {description}`
- 示例：`feat: add product application create page`

### 7. 开发总结
- 每轮开发完成后，在 `docs/development-summary/` 下新增开发总结文档

### 8. 每日收尾
- 当天结束后，在 `docs/daily-reports/` 下新增或更新项目日报
